# 古文阅读管线

## 工作流

### 模式A：单篇精读（用户点读一篇）

#### 第一步：获取原文

用户发ctext.org链接 → 用**curl**抓取（Python requests会被CAPTCHA拦截，必须用curl -sL）。

⚠️ **关键陷阱：ctext.org有反爬保护，多次请求会触发人机验证。第一次请求必须成功拿到全部内容，后续解析全部在本地完成。绝不要在解析中途再发第二次请求。**

```bash
curl -sL "https://ctext.org/wiki.pl?if=gb&chapter={chapter_id}&remap=gb"
```

原始处理命令（一次性完成，不再发第二次请求）：
```bash
curl -sL "URL" | sed 's/<br[^>]*>/\n/g' | sed 's/<[^>]*>//g' > /tmp/raw_article.txt
```

每卷包含多篇文章，用 `《文章名》` 字符串搜索定位。

#### 第二步：提取目标文章

从HTML中提取纯文本（全部在本地操作，不再请求ctext）：
1. 将`<br>`替换为换行
2. 去除所有HTML标签  
3. 定位文章起始标记（`《文章名》`）和结束标记（下一篇的`《文章名》`）
4. 保留原文+注释，不丢弃inline注解（那是古籍自带注释）
5. 用Python字符串查找法（text.find()）精确截取，不用awk/sed模糊匹配（中文锚点可能因转码丢失）

#### 第三步：三层分析

用户确认的产出格式——**严格按此顺序**，不可缺层：

#### ① 原文精华
- 精选原文最值得品味的段落/句子（3-5句）
- 附字数统计（如「281个字讲完一场战争」）
- 标注原文，不加翻译（用户要的是原文本身）

#### ② 文章心法（文学角度）
- 结构分析——篇章布局、起承转合
- 笔法赏析——修辞、节奏、对比
- 左氏/作者的评论手法（如有「君子曰」类结尾评语）
- 表格对比是最佳展示形式（阶段→核心问题→回答）

#### ③ 管理映射（框架对话）
- **必须**绑定用户现有管理框架对话，不做纯文学分析
- 优先对接：任正非针尖式压强/危机转化/开放吸收 → 五步转化管道（拆连筛测沉） → 六韬知行哲学 → ZN-47华为操作机制框架
- **表格对比是最佳展示形式**——列出管理框架 vs 文章做法的对照
- 对用户的直接启发（「这对你的包打听/异虎团队场景意味着什么」）
- 已吸收的管理框架要引用具体ZN笔记号（如 ZN-46、ZN-47、ZN-06）
- 启发部分必须绑定用户业务场景（包打听/异虎团队/FDE），不做泛泛的「人生启示」

#### ④ 用户偏好
- 每篇开头**不要翻译原文**——用户要的是原文本身，不是白话翻译
- 篇幅控制在合理范围内（500-800字分析 + 原文引用），不堆砌
- 如果用户没提出异议，分析完成后**主动问**「是否沉淀入库」
- 入库后的确认信息要列成**管线跑通确认表**（四列：步骤、状态、产物）

### 模式B：批量处理（用户说「第一批搞完」）

当用户要求一次处理多篇文章时，使用 `delegate_task` 并行化：

1. **分批策略**：每批不超过8篇，用 `delegate_task(tasks=[...])` 每次最多3个并发子代理
2. **分卷原则**：按原文卷次分组（卷一左传一批、卷四秦文一批等），同一子代理处理同一卷，避免跨卷混乱
3. **每子代理的任务结构**：给子代理传递每篇的以下信息：
   - 原文关键句和梗概（子代理不需要自行找文本，除非没有现成文本）
   - 管理映射要点（针尖式压强/五步管道/包打听等）
   - 编号分配（如 ZN-49 到 ZN-53）
   - 文件路径和目标文件名
4. **子代理工具集**：`["file"]`（只写文件足够）或 `["file", "terminal", "web"]`（需要找文本时）
5. **避坑**：子代理写文件时路径必须带 `/mnt/c/Users/18502/...` 的绝对路径；文件内容要独立完整，不依赖子代理间通信
6. **完成后**：统一更新 ZN-00 总纲（见第四步-批量模式）

### 第四步：沉淀回库

用户确认后才执行。**先问，不问就干会踩线。**

#### 单篇模式

ZN笔记创建规则：
- **命名**：`ZN-序号-文章名-管理视角.md`
- **序号**：先 `ls "/mnt/c/Users/18502/Documents/Obsidian Vault/六韬智脑/ZN-"* | sort` 确认最大号
- **路径**：`/mnt/c/Users/18502/Documents/Obsidian Vault/六韬智脑/`
- **结构**（严格按此顺序，不可缺层）：
  1. `# 文章名 · 管理视角` 标题
  2. 元信息块：原文、作者、字数、一句话定位
  3. `---` 分隔线
  4. `## 原文精要` — 精选段落（3-5句）+ 字数统计
  5. `## 文章心法` — 文学/结构分析，多用表格
  6. `## 管理映射` — 与已有框架的对话
  7. `## 对[产品/场景]的启发` — 对用户业务场景的直接映射
  8. `## 关联笔记` — `[[wikilink]]` 格式，至少关联 ZN-06 孙子兵法、ZN-46 任正非、ZN-47 华为
  9. `## 元数据` — 创建日期、来源、类型、标签、入仓评分

**入仓评分公式**：源典性30% + 密度25% + 参考频次20% + 跨模块度15% + 体量10%，每项1-5分，总分 /5

## 古籍全文库模式（全文检索 vs 精读批注）

古籍处理分两条线，**不要混淆**：

### 双轨策略

| 维度 | 全文检索库（古籍全文/） | 精读批注（ZN-笔记） |
|------|----------------------|------------------|
| 目的 | 可检索的原文基底 | 管理框架弹药 |
| 内容 | 纯原文，不掺分析 | 三层分析+框架对话 |
| 位置 | `古籍全文/` 子目录 | `六韬智脑/ZN-*` |
| 字数 | 几千到几十万 | 2-6KB/篇 |
| 适用 | 用户需查原文/引经据典 | 用户需讨论/沉淀管理思想 |

### Token意识策略（来自用户偏好「悠着点搞」）

| 文本规模 | 策略 | Token消耗 |
|---------|------|:---------:|
| <5K字（如心经、孝经） | write_file直接写全文 | 极低 |
| 5K-50K字（如论语16K字） | delegate_task子代理写 **或 execute_code 直接写入** | 中等 |
| >50K字（如史记500K字） | **不写全文**，建索引页+TOC+外链 | 极低（~5KB索引） |
| >500K字（如资治通鉴3M字） | 仅索引页+外链 | 接近零 |

### 批量写入技巧（execute_code 模式）

当需要一次性写入多部古籍原文时，`delegate_task` 子代理容易因网络/超时而失败。更可靠的方案：

1. **直接在 SKILL 会话中用 `execute_code` 写**，不用 `delegate_task`
2. **写法**：在单个 `execute_code` 内连续调用 `write_file()` 写入多部文本
3. **内容取舍**：不是每部都要写完整原文——
   - **核心典籍**（易经、道德经、论语等）：写完整原文
   - **大型文集**（唐诗三百首、诗经等）：写代表性篇章（风雅颂代表篇目/各体裁代表作）
   - **哲学论著**（孟子、庄子、荀子等）：写内篇/核心篇章全文，外篇存目即可
   - **篇幅极长**（史记、资治通鉴等）：只写索引页
4. **优先排序**：按用户的兴趣优先级（历史P0 > 易经P1 > 命理中医P2）决定哪些写完整、哪些写节选
5. **文件头**：每篇写 `# 书名 全文` 和 `来源：通行本《书名》`，即使只是节选也要让使用者能快速理解文本性质
6. **避坑**：`write_file` 路径必须用绝对路径（`/mnt/c/Users/...`）；一次 `execute_code` 中串行写入多部不会冲突

**大文本索引页模板**（用于古籍全文库）：
```markdown
# {书名}

> **作者**：{作者}  
> **字数**：~X万字  
> **来源**：外部链接索引页  
> ⚠️ 全文未本地存储

**全文阅读**：{外部链接}

## 目录（{篇数}篇）
{表格化TOC：编号+篇名+内容摘要}
```

### 古籍全文库文件结构

```
古籍全文/
├── README.md          ← 索引总表（含状态标记）
├── 道德经.md          ← 小文本，完整原文
├── 论语.md            ← 中文本，完整原文（约63KB）
└── 史记.md            ← 大文本，仅索引页+外链
```

所有文件在 `古籍全文/README.md` 中维护索引总表，按「已入库/待入库」分类，每个文件含元数据块（作者、成书时代、字数、来源、入库日期）。

## 外部文本源链路（ctext.org被封锁时的替代方案）

| 来源 | 可用性 | 适用场景 |
|------|--------|---------|
| ctext.org/wiki.pl | 首次✅，后续❌（CAPTCHA） | 一次性拉取多篇文章 |
| ctext.org/library | 可能可用 | 备选路径 |
| zh.wikisource.org | 从WSL不可达 | 备用 |
| raw.githubusercontent.com | 不稳定 | 备用 |
| **训练知识重建** | ✅ **可靠** | **首选方案：小中型文本（<50K字）直接用write_file从训练数据重建** |
| 用户提供的链接/文件 | ✅ 最可靠 | 优先使用用户给的材料 |

**核心原则**：短文本（<50K字）直接从训练知识完整重建，不依赖外部网络。仅大文本需要外源。

## 注意事项

- ctext.org内容有inline小字注释（释词/背景/点评），非常宝贵，不要丢弃
- 每篇文章末尾有「君子曰」或总结评语，属于文章心法的一部分，必须保留
- 用户偏好：原文入仓偏好实用决策类（管理/历史/战略强相关），不选纯抒情/纯文学炫技篇目
- ctext.org CAPTCHA 策略：首次curl可成功获取，后续同一IP的请求会被拦截。必须在第一次请求中拿到所有需要的内容。
- 子代理的文件路径必须用 `/mnt/c/Users/18502/Documents/Obsidian Vault/六韬智脑/` 绝对路径，不要用相对路径
- 用户说「悠着点搞」时，优先考虑token消耗而非完整性——大文本只建索引不存全文
- 全文入库和精读批注是两条独立管线，不要在一次任务中混用
