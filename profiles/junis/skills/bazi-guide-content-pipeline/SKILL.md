## 1. 11维度预创作分析（内部思考，不展示给用户）

每次创作前，AI在内部完成以下11个维度的系统思考。这是Viral Writer的核心方法论，区别在于：**维度内容必须基于八字命理的垂类知识，不是通用模板**。

### 1.1 核心观点
**原则**：一篇文章/帖子只需要一个锐利的观点。
**八字场景**：不是"八字是什么"（太泛），而是"为什么你的出生时间决定了你的决策模式"（具体、可讨论）。
- 好：你的五行缺什么，比你的星座更基本
- 差：八字入门介绍

### 1.2 副观点（2-3个）
让核心观点立体，但不抢主角：
1. 每个元素对应的人格特质（五行→性格映射）
2. 为什么西方占星和东方八字看待"命运"的方式不同
3. 实际案例：某个元素过旺/过弱的真实影响

### 1.3 说服策略
选择最适合的1-2种：
- **类比隐喻**（八字=基因代码、五行=操作系统）— 八字内容首选
- **故事驱动**（客户真实案例，脱敏后）
- **数据驱动**（统计比例、元素分布）
- **权威背书**（古籍《三命通会》、经典理论）
- **社会认同**（"越来越多人在探索"、"1M+用户已尝试"）
- **反常识**（"你以为的幸运年，可能恰恰是你最需要谨慎的"）

### 1.4 情绪触发点
选择1-2个：
- **好奇** — "你的八字里藏着什么秘密？"
- **共鸣** — "为什么你总觉得某些年份特别难？"
- **优越** — "大多数人只知道星座，而你在了解更高级的体系"
- **归属** — "欢迎来到超越12星座的古老智慧世界"

### 1.5 金句（2-3个）
能脱离上下文独立传播的记忆锚点：
- "你的生辰不是一个日期，是一组密码"
- "五行不是玄学，是古代中国人的系统论"
- "你不是运气不好，只是你的火克了你的水"

### 1.6 情感曲线
设计正文的情绪起伏：
```
开头（好奇/共鸣）→ 中段（认知冲击/知识点）→ 结尾（平静启发/行动召唤）
```

### 1.7 情感层次
- **表层**：有趣的知识点（"原来五行生克是这个意思"）
- **中层**：自我认知（"这解释了我为什么总是..."）
- **深层**：身份认同（"我在了解比星座更古老的智慧"）

### 1.8 论证多样性
交错使用，避免单一：
- 故事 → 数据 → 类比 → 反问 → 经典引用

### 1.9 视角转化
- **英文**：第二人称为主（"You"），拉近距离
- **中文**：第一/第二人称交替（"你、我们"）
- 不长时间用第三人称（太像教科书）

### 1.10 语言风格
- **英文**：简洁、有力、短句为主。Playfair Display标题用经典感，正文Inter用现代感
- **中文**：有文言质感但不过度，一个句子不超过25字
- **禁止**：学术腔、翻译腔、过于华丽的修辞
- **节奏**：短句（冲击）→ 中句（解释）→ 短句（收尾）

### 1.11 互动钩子
让读者忍不住回复的设计：
- **提问**："Which element do you feel most drawn to?"
- **填空**："My BaZi chart says I'm ____-dominant"
- **选择**："Are you ⛰ Earth or 🌊 Water?"
- **挑战**："Guess your element in the comments, I'll confirm"

## 4. 排版规则（VIS规范）

1. **80/20留白**：内容不超过画面20%
2. **黄金偏移**：文字放在左下1/3或右下1/3，永不居中
3. **三层结构**：背景层(纯色) > 中间层(微噪点纹理) > 前景层(文字)
4. **金线分割**：0.5-1px Raw Gold细线
5. **禁止居中排版**

### Pitfalls

- ⚠️ 用户明确要求：直接发 `.md` 文件，不在聊天里铺内容
- ⚠️ 首次使用此skill时，先确认当前的排期频率（每日 vs 每周）和相关cronjob是否存在
- ⚠️ 夏令时（PDT/PST）变化影响UTC偏移，排期用当地钟表时间（如"晚7点 PT"）而非UTC绝对值
- ⚠️ **两阶段管线盲区**：发布稿（`八字出海-发布稿-*.md`）由鲁班9:00生成（deliver=local），燕青9:50脚本负责上传到云盘。如果修改了upload脚本或改了发布稿文件名/路径，必须同步更新 `daily-bazi-upload.py`。用户不会收到「发布稿上云了吗」的提醒——只有投诉时才知道没传上去
- ⚠️ **bot上传不可见陷阱**：bot身份上传的文件用户默认不可见。上传脚本必须内置 `drive permission.members create` 授予用户 `full_access`。每次新增文件类型（如发布稿）都要检查授权逻辑是否覆盖到
- ⚠️ **批量生成格式不一致陷阱**：用 `delegate_task` 并行生成多天发布稿时，子代理各自按自己的理解输出格式，容易产生不一致。**解决方法**：在 delegate_task 的 context 中嵌入格式规范全文（允许的4个章节、禁止章节列表、Prompt双引号规则），并明确指向 `references/canonical-publishing-draft.md` 为格式标杆。生成完毕后必须抽查头尾文件的一致性
- ⚠️ **重传不清旧版陷阱**：用 `drive +upload` 重新上传同名文件到云盘时，飞书不会覆盖旧文件，而是新增一个副本。**所以重传后必须主动删除旧版本**：`drive files list` 确认重复文件 → `drive +delete --file-token <旧token> --type file --yes` 删除旧版。详见 `references/drive-delivery-automation.md`
