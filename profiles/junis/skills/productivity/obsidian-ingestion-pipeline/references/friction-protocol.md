# 摩擦协议（Friction Protocol）

> 吸收自 GBrain 的 `skills/_friction-protocol.md`（Garry Tan）

## 核心理念

任何让用户感到困惑、文档与工具不一致、找不到下一步的地方——都是**一等信号**，记录下来而不是忍着。

## 何时触发摩擦日志

在执行以下任务时，注意识别这些信号：

| 信号 | 示例 |
|:----|:------|
| **命令/操作报错但错误信息无帮助** | curl 超时后只返回空字符串，不告诉你为什么 |
| **文档说一套，实际做一套** | 文档说用 `npx` 但实际需要先 `npm install` |
| **找不到下一步** | 执行完 Step 3 后，文档说"视情况继续"但不给判断条件 |
| **需要用户手动弥补工具不足** | 知乎文章 curl 抓不到，最终让用户手动复制 |
| **操作成功但无法验证** | crontab 添加成功但没有确认输出 |

也可记录正面信号（Delight）：
- 某个工具/命令第一次就成功了，文档完全准确
- 错误信息直接给出了修复命令
- 猜出来的 flag 名称恰好存在

## 如何记录

### 即时记录（在当前会话）

当摩擦发生时，在回复末尾追加一行：

```
🔴 摩擦：<一句话描述>
💡 修复：<如果已经有>
```

### 持久记录

如果同一摩擦出现 2 次以上 → 按三振晋升规则，写入 `obsidian-ingestion-pipeline` 的 pitfall 列表或创建 reference 文件。

如果仅出现 1 次但很严重（阻塞了任务）→ 直接写入 `~/.hermes/resources/frictions/YYYY-MM-DD.md` 留档。

## 例子

```
🔴 摩擦：知乎文章 curl 返回 zse-ck 反爬页面，全量抓取方案均失败
💡 修复：需用户手动复制粘贴，或等搜索引擎缓存
⏱ 后续：如果同一网站反复出现 → 在喂料流水线 pitfall 中加一条"知乎/雪球类平台需手动"
```

```
✅ 愉悦：deep-research 流程走完一遍后，每一步的中间结果展示节奏恰好，用户无纠正
💡 可复用：研究型问题的这个交互节奏可以固化
```

## 与三振晋升的关系

| | 摩擦协议 | 三振晋升 |
|:---|:--------|:--------|
| 触发时机 | 单次摩擦发生时 | 同一问题出现 ≥3 次后 |
| 记录方式 | 日志行 / 临时文件 | 写入 Skill 或 Obsidian |
| 目的 | 不遗忘、可追溯 | 永久修复、不再重犯 |
| 关系 | 三振晋升的信号源 | 摩擦协议的最终归宿 |
