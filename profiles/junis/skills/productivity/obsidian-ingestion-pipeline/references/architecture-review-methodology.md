# 知识库架构评审方法论

> 当需要重新审视知识库的底层架构、模块划分、命名体系或总纲结构时，使用此方法论。
> 
> 这是一个从"提案→评审→调分→执行→打标"的完整闭环，确保每次架构变更都有据可查、有分可评、有迹可循。

---

## 工作流

```
  燕青提案（自评分数）
       ↓
  Hermes交叉评审（独立打分 + 指出矛盾）
       ↓
  调整方案（修正逻辑矛盾 + 目标分 > 9）
       ↓
  用户确认（"go" 或反馈）
       ↓
  分阶段执行（Phase 1/2/3，按风险分级）
       ↓
  更新提案文档（记录实际评分 + 完成状态）
```

## 六维评分体系

每个维度按 1-10 打分，加权求和得总分。

| 维度 | 权重 | 含义 | 扣分典型信号 |
|:-----|:----:|:------|:------------|
| 框架精确度 | 30% | 核心逻辑是否自洽，有没有自相矛盾 | 定位矛盾（如"既是A又是B"）、流程断点 |
| 命名一致性 | 20% | 术语是否互斥，模块边界是否清晰 | 两字含义重叠("决≈断")、同一概念分身两处 |
| 内容可承载性 | 20% | 方案能否容纳现有全部内容，不丢不混 | 漏分类、误合并(把不同笔记当分身)、分组未落地 |
| 使用友好度 | 15% | 用户进入知识库能否快速找到方向 | 入口模糊、场景速查缺失、树结构过深 |
| 可维护性 | 15% | 后续增量是否可持续 | 需要搬文件、手工维护多处、不搬文件但维护链条过长 |
| **总分** | 100% | 加权求和，目标 > 9.0 | — |

### 评分锚点参考

| 分数 | 含义 |
|:----:|:------|
| 10 | 完美，无需调整 |
| 9.0-9.5 | 优秀，有1-2个可优化的边缘细节 |
| 8.0-8.9 | 良好，有1-2个需要修正的逻辑矛盾 |
| 7.0-7.9 | 及格，存在结构性缺陷 |
| < 7.0 | 需要重来 |

## 常见架构矛盾模式（经验积累）

### 模式1：定位矛盾
**症状**：同一个组件在方案中既是"A"又是"B"，或在树结构中处于矛盾位置。
**案例**：太极双螺旋既是"元认知操作系统(道)"又被压缩为"观(信息摄入)"步骤 → 修正为地基层

### 模式2：分身误判
**症状**：将两篇不同内容/不同深度的笔记误认为同一个内容的分身，建议合并。
**案例**：物我合一在智脑(方法论框架)和易哲(哲学思考)各有一篇 → 确认是不同笔记，标注"双版本"

### 模式3：同源误判
**症状**：同一题材的不同处理深度被误认为是"分身"。
**案例**：百年孤独在史鉴(深层哲学分析11KB)和易哲文史哲(阅读笔记)各有一篇 → 不是分身，是"深层分析正本 vs 阅读参考"

### 模式4：分组未落地
**症状**：方案说"分为X组"但没有体现具体哪些文件归哪组。
**案例**：四分组说"决策框架/战略/AI/人物"但29篇笔记未映射 → 每篇精确分配到象限

### 模式5：包裹vs地基
**症状**：元认知层的定位——是包裹全局还是底层地基。
**规则**：用户的核心哲学是"地基"而非"包裹"——太极双螺旋是最底层的思想范式，上层建筑建基其上。

### 模式6：编号冲突
**症状**：同一编号被多篇笔记使用（跨模块或同模块内）。
**2026-05-27 已修复**：全库 AA-NN-模块前缀体系，前缀使每个模块独立计数。

### 验证清单

```bash
# 旧文件名不存在
find "$VAULT" -name "旧文件.md" 2>/dev/null  # → 空

# 新文件存在
test -f "$VAULT/模块名/新文件.md" && echo "YES"

# 无残留右编号（NN- 格式）
find "$VAULT" -maxdepth 2 -name "[0-9][0-9]-*.md" ! -path "*/01-信息流/*"
```
