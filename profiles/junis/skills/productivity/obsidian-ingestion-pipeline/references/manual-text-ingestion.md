# 手动物料入库模式（Manual Text Ingestion）

## 场景

用户提供**原始文本/手动转录/口述转述**，而非可抓取的链接或PDF。典型触发：

- 视频号文案录播 → 用户听完手动转录成文字发过来
- 用户自己写的笔记/心得 → 要求结构化入库
- 语音转文字后的草稿
- 书摘/金句合集

## 与标准流程的区别

| 维度 | 标准流程（自动抓取） | 手动录入模式 |
|:----|:------------------|:-------------|
| 输入来源 | URL/PDF → api提取 | 用户发来的原始文本 |
| 提取阶段 | 跳过（已有文本） | 跳过（已有文本） |
| 核心工作 | 分类+注入 | 结构化+分类+注入 |
| 质量 | 受源网页质量影响 | 用户已过滤，质量高 |
| 依赖 | 需网络/OCR/抓取工具 | 零依赖 |

## 结构化方法

用户提供的原始文本通常是**流水账式叙述**（如听完整段视频后口述/打字），需要做三层转换：

### 第一层：分段识别

识别用户的原始文本中的自然分段信号：

```
原始："第一章 为什么不能只用默认模型...好多人刚开始会直接选一个看起来最强的模型...第二章 综合优先级量化表..."
    → 识别出章节标记（第一章/第二章/一/二/数字序号）
    → 拆出章节标题和正文区域
```

分段信号优先级：
1. 显式标记：`#` `一、` `第一章` `Chapter 1` 
2. 自然分段：用户换行/带引号/口语化的分隔词（"所以今天这期视频" "最后我们总结一下"）
3. 无信号（纯流式文本）：靠主题变化自然切分

### 第二层：信息分类

原始文本中的信息类型需要区分对待：

| 类型 | 示例 | 入库形态 |
|:----|:-----|:---------|
| **框架性描述** | "这张表把模型能力拆成几个核心维度" | 表格 / 清单 |
| **规则/原则** | "高价值任务优先稳定性" | 加粗标题 + 正文 |
| **数据/排名** | "K2.6排在最前面，GLM-5.1紧随其后" | 表格行 / 编号列表 |
| **对比/决策** | "不推荐优先使用MiniMax-M2.5或Hy3preview" | 「不推荐」列 / 警告框 |
| **总结性语句** | "记住一句话WB不是选最强而是选最合适" | 核心命题引用块（> 格式） |
| **过渡/填充词** | "所以今天这期视频我们要讲的..." | 删除（转成章节标题和正文） |

### 第三层：格式增强

| 原始文本形态 | 增强为 |
|:-----------|:-------|
| "90~100分是高价值任务主力池" | `### 🏛️ 高价值任务主力池 (90~100分)` 表格 |
| "第一类复杂代码项目首选GLM-5.1和K2.6" | `## 场景① 复杂代码项目` + 优先级表格 |
| "记住一句话" | `> **核心观点：** ...` 引用块 |
| 模型列表 | `| 模型 | 定位 | 最佳场景 |` 三列表格 |

## 文档结构模板

用于手动转录→知识库入库：

```yaml
---
tags: [主模块, 子标签]
aliases: [中文别名, 英文名]
created: YYYY-MM-DD
source: "微信视频号 · 用户转录"
---
# 标题

> 一句话核心观点

## 一、概述（可选）

## 二、核心框架（必选）
用表格/清单呈现核心分类或体系

## 三、详细展开（可选）

## 四、总结
一句话收尾 + 行动号召

*来源：用户转录自视频号内容，由国服第一模型整理*
```

## 关键原则

1. **保留核心信息，删除口语填充** — 用户的转录可能包含"好""那""其实""也就是说"等口头禅，删除不影响信息的
2. **尊重用户原意，不改框架** — 用户自己提炼的知识框架必须原文保留，不重新措辞
3. **添加源标注** — 底部注明来源
4. **补充可操作性** — 用户给了理论框架，主动补一列实操建议
5. **优先表格** — 多个同类项用表格呈现

## 实测案例

| 素材 | 原始形态 | 入库产物 | 行数 |
|:----|:--------|:--------|:---:|
| WorkBuddy模型路由教程 | 用户手动转录6章视频文案（~3000字流水） | 结构化笔记 + 模型路由脚本 + WB技能 | ~200行 |
