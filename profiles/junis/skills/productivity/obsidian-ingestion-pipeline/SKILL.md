### WB→Vault 批量导入流程

当用户要求从WB导入原文时，遵循此顺序：

**Step 1: 优先级判定**
按用户兴趣方向排P0→P1→P2：
- P0: 易经新方向（梅花易数、增删卜易、黄金策、卜筮正宗、易经白话等）
- P1: 历史识人（冰鉴 → 六韬史鉴/典籍/）

**Step 2: 格式处理（期刊排版格式规范）**
每个文件必须含：
```yaml
---
title: {书名}
author: {作者}
tags:
  - 六韬易哲
  - {模块名}
  - {分类}
category: 典籍
type: fulltext
source: 六韬易哲技能·WB
---
```
- 中文标点全角化（，。、？！：；「」《》）
- 段落间距标准化（paragraphs之间空一行）
- 末尾加 `*返回 [[六韬易哲]]*`
- 用脚本批量处理（≥3个文件必须用Python脚本）

**Step 3: 链接建立（三步链）**
1. 更新 `module.md`（如易经体系.md）→ 笔记列表 + 关联知识段
2. 更新 `index.md` → 摘要列表
3. 更新 `00-六韬总纲.md` → 统计数字

## 处理步骤（完整工作流，已验证7单）

0. **预检今天完整素材清单** — 用户说「把今天新增的都融入大脑」时，必须回溯今天完整会话，不能只处理最后一条消息：
   - 用 `session_search(search="今天 新增 文章 分享", limit=5)` 查询当日全部会话历史
   - 检查 `01-信息流/` 中 `status: raw` 的素材
   - 检查飞书 Bitable 收藏随想录表是否有未入库的今日记录（来源类型: 聊天记录/文章）
   - 汇总清单后一次性处理，避免用户二次纠偏「还有之前的哦」
1. **收件箱登记** — 在 `01-信息流/` 创建素材笔记，填入类型/标题/源文件，标记 `status: processing`
2. **提取全文** — 用 pymupdf 提取 PDF 全部文本。**先读目录+前言**把握整体结构（5-10页），再读关键章节
3. **⚠️ 扫描PDF检测** — 如果 pymupdf 提取结果 < 50 字/页，则为图片扫描版。
   - **先做源发现（Step 0）**：用 `strings` 检查PDF中是否有嵌入的URL/元数据。许多PDF是通过Chrome"打印为PDF"生成的，原文可能在网页上可直接抓取——这比OCR省上千倍token。详见 ocr-and-documents 技能的「Step 0: PDF Source Discovery」。
   - **先提取可用元数据**：如果网页也抓不到（WAF/登录墙），很多扫描版PDF的水印/页脚中仍有少量可读字符（如"三个皮匠报告站 文档Id:1191225"等）。用这些碎片——来源网站URL、文档ID、碎片关键词——帮助用户定位文字版。
   - 标记 `status: raw` 并告知用户需要文字版，附上提取到的元数据
4. **分类判定** — 根据内容和类型，决定主模块+副模块。参考 `references/classification-examples.md` 的真实案例
5. **单源双抽出** — 如果素材同时有**哲学/历史根基（道）**和**实用方法论（法）**两个层面的价值，创建两篇并列笔记。
   - 例：黄奇帆→智脑10+史鉴10（分析框架 vs 一手史料）
   - 例：品胜→智脑11+实战案例索引（方法论 vs 案例库）
6. **姊妹篇检测（∃ companion？）** — 同一作者/同一主题的后续作品，必须链接到已有笔记。
   - 例：黄奇帆《战略与路径》→ 链到 [[10-黄奇帆经济分析框架]]，标注"姊妹篇，第一卷→诊断，第二卷→路径"
   - 原则：同作者的后续作品提炼"增量"而非"重复"——只提炼第一卷没讲的新概念/新框架
7. **Web文章提取（非PDF素材）** — 对于文章链接：
   - **复杂页面（视频号/SPA/JS渲染）优先查阅 `productivity/web-content-extraction` skill**（含 Trafilatura / Jina Reader / Playwright / CDP / Bing搜索 等多策略体系）
   - 一般微信公众号文章直接用以下 curl 方案抓取：
   - 用 `curl -sL --max-time 30 -A "Mozilla/5.0 (Linux; Android 12; Pixel 6) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/120.0.0.0 Mobile Safari/537.36"` 抓取HTML（微信需要Mobile UA，否则重定向验证页）
   - **微信文章优先方案**：提取 `id="js_content"` 的 div 内容后去标签，比正则更精准（保留段落结构、不乱序）
   - **⚠️ 视频号链接识别**：`weixin.qq.com/sph/` 路径前缀是**视频号（WeChat Channels）**链接，不是公众号文章。页面为JS动态渲染的视频页面，curl 抓取后只有框架代码、无正文内容。**难以通过常规 Web 提取获取文案**（Jina Reader 超时、CDP端口不通、可能需微信登录态）。处理方式：\n     - 如果用户同时提供了引用语/截图，以用户提供的文字为准\n     - 尝试用 vision_analyze 读取截图（如果当前模型支持）\n     - 搜索 Bing 找同一内容（用核心语录作为搜索词）\n     - **截图OCR提取**：如果用户发了长截图且当前模型不支持视觉，用 RapidOCR（ONNX Runtime）本地OCR提取中文文案。详见 `references/rapidocr-setup.md`\n     - 最终手段：告知用户这是视频号内容、无法自动提取，请用户提供文案文字版
     ```python
     m = re.search(r'id="js_content"[^>]*>(.*?)</div>\s*<script', html, re.DOTALL)
     if m:
         text = re.sub(r'<[^>]+>', '', m.group(1))
         text = html.unescape(text)
     ```
- **备用方案**（非微信页面）：Python 正则 `re.findall(r'[\\u4e00-\\u9fff\\u3000-\\u303f\\uff00-\\uffef]{20,}', html)` 提取中文段落
- **备用方案二（Python/bs4 超时时的对冲）**：如果 Python 的 bs4/lxml 环境异常导致提取命令超时（表现为 even `python3 -c "from bs4 import BeautifulSoup"` 卡死），不要反复重试。改用 `read_file` 工具直接扫描 HTML 文件，在原始 HTML 中人工定位文章内容：
  1. 先用 `search_files` 在 HTML 中搜 `entry-content`、`content-inner`、`article`、`post-content` 等典型 WordPress 文章容器 class
  2. 定位到内容 `div` 的起始行号，用 `read_file` 读取该区域（200-500行）
  3. 在 HTML 段落中直接用肉眼提取 `>文本<` 内容，手动整理为 markdown
  4. 常见 WordPress 主题的内容容器模式：
     | 主题 | 容器选择器 |
     |------|-----------|
     | jnews（電腦王阿達等） | `div.entry-content.no-share > div.content-inner.jeg_link_underline` |
     | 一般 WordPress | `article .entry-content` 或 `.post-content` |
  5. 注意过滤广告 div、相关推荐（`yarpp-related`、`jp-relatedposts`）、评论区等非正文区域
  6. 此方法不依赖 Python 环境，纯工具操作，适合 WSL 环境
  注意：`js_content` 提取的全文很大（3MB HTML 中正文约3-5K字），需截断或只取头50行预览
  若遇WAF（雪球、三个皮匠报告站等平台），标记 `status: raw` 并告知用户需手动复制内容。见 Pitfall #6。
8. **提炼笔记** — 创建内容笔记而非内容摘要：
   - 书籍：提炼为"可迁移的分析框架/方法论"
   - 报告：提炼为"诊断工具/评估模板"
   - 管理类：提炼为"操作原则/流程模板"
   - 正文用 ## 分章，### 分节，编排表格/清单结构
9. **建链** — 使用 `delegate_task` (子代理) 同时创建主+副笔记。每条笔记链接到至少 3 篇现有笔记
10. **更新总纲三处** — `00-六韬总纲.md` 的三个区域都要更新：
   - **模块地图（树形图）** — 新增笔记按「树」结构插入：根系（双螺旋）/ 主干（史鉴）/ 枝干（智脑·易哲）/ 枝干交汇（决断之桥）/ 树冠（实战案例）。新建双螺旋模块笔记可能属于「破土成枝」（根系直接长出，不走主干），标注在双螺旋🌳根系的子列表中
   - **使用场景速查表** — 添加新素材的入口场景
   - **核心理念区** — 如新素材改变了结构认知
11. **更新收件箱** — 标记素材 `status: processed`
12. **可视化交付** — 生成拓扑图 SVG
13. **反向链接注入（回链）** — 新笔记创建后，检查其他笔记是否链接回它：
    - 在源文件的 `📖 关联阅读` 区加一行 `- [[新笔记名]] — xxx`
    - 在对比/对标/对照段落插 `对照样本：XX的[[新笔记]]提供了YY对照`
    - 验证：`search_files(pattern="新笔记名或路径")` 确认至少 2 个其他笔记有链接回来
    - ⚠️ 缺少回链的后果：Obsidian 图谱中笔记显示为孤立节点
**14. 判断是否走深度模式** — 按下方「深度模式（Deep Mode）」章节判断素材深度。思想/哲学/方法论/传记类素材直接走深度挖掘（v2），整书 200页+ 走大型素材并行拆解（模式4）。纯信息类/报告/数据类走 v1 标准提炼，跳过此步。
15. **呈报用户** — 用表格+图展示完成清单
16. **归档原始素材（`_raw_sources/`）** — 处理完成后，将原始素材（文章原文/PDF全文/截图等）保存到 `_raw_sources/` 目录作为不可变层（Immutable Layer），灵感来自 Karpathy LLM Wiki 的 raw/ 设计理念：
    - 文章/网页 → `_raw_sources/articles/<文件名>.md`，含 frontmatter（source_url, ingested, type）
    - PDF/论文 → `_raw_sources/papers/<文件名>.md`
    - 图片/截图 → `_raw_sources/assets/<原文件名>`
    - 信息流笔记的 frontmatter 新增 `raw_source` 字段指向归档文件
    - **原则**：不碰 `_raw_sources/` 下的文件。更正/更新写在知识库笔记里。溯源时读 `raw_source` 字段。

17. **Skill 双通道同步（双通更新）** — 当素材是关于某位**已有关联 Hermes Skill 的人物**时（如段永平→duanyongping-perspective、马斯克→musk-cognitive-framework），在 vault 笔记更新后，必须**同步更新对应的 Skill**。这确保知识库和角色扮演能力保持一致：
    - **检查**：`skill_view(name)` 查看是否存在对应的 Skill
    - **Skill 更新内容**：新增核心模型（如营销心法、教育观等全新话题）；更新已有模型（如平常心定义链的升级版本）；增补金句/口头禅（25年访谈新增的口语表达）；修订过时信息（如黄峥"门徒"标签的澄清）；更新角色扮演模板中的新案例
    - **技巧**：Skill 中使用编号模型（模型1、模型2…）时，如在中段插入新模型注意重新编号，或用标题锚点替代编号避免断号
    - **验证**：`skill_view(name)` 确认新内容可见，无断号/orphan 内容

### 产出纪律（No Slop — 吸收自 GBrain _output-rules.md）

喂料产出的笔记是**持久知识工件**，不是聊天记录。以下规则必须遵守：

| 规则 | 说人话 | ✅ 好 | ❌ 不好 |
|:----|:------|:-----|:-------|
| **没有填充词** | 不写"值得注意的是…"、"有趣的是…"、"不可否认…" | 直接给事实 | 值得注意的是，这个框架有三大支柱… |
| **有引用的不模糊** | 引了来源就不说"可能" | 根据何庭波论文，τ缩放将系统延迟从微秒级降至~100纳秒 | τ缩放可能能降低延迟… |
| **没有 AI 开场白** | 不提"我为你整理了…"、"以下是…" | 直接写标题或总起句 | 我为你整理了这篇笔记… |
| **没有占位日期** | 不用"近日"、"最近"、"在不久的将来" | 明确写日期或事实 | 近日这个领域取得了突破 |
| **短段落实体事实** | 一段一个事实，附引用 | 每段 2-4 句 | 一大段 200 字无结构 |
| **原话照录** | 别人的思考用他的原话，不改语法 | > "做对的事，然后把不对的事从 stop doing list 上划掉" | 段永平认为应该做正确的事并减少错误 |
| **标题质量** | 足够描述性 + 不超过60字 + 不是句子 + 不是泛称 | ZN-32-华为韬定律·时间缩放理论 | ZN-32-一个关于芯片发展的理论 |

**底线测试**：写完一条笔记后问自己——"如果一年后回来读，这还是一篇高质量笔记吗？" 如果是，符合标准。如果感觉像一篇聊天摘要，重写。

### 主动发现应用层链接（Real-world Application Domain Scanning）

当笔记内容是**实践型/策略型知识**（兵法、管理、战略、权谋类）时，不能只链到思想根源笔记——必须**主动扫描三个应用维度**并建立链接：

1. **战略层**（谁在最高决策层面用过？）→ 如毛泽东对孙子兵法的应用 → 链到[[DS-05-毛选]]
2. **战术层**（谁在执行层面用过？）→ 如刘伯承对孙子兵法的战术运用 → 链到[[DS-11-邓小平时代]]
3. **商业层**（谁在现代商业中暗合此原理？）→ 如华为/拼多多/特斯拉的商业策略 → 链到[[DS-10-改革开放]]

**判断方法**：读笔记核心内容后自问——「这些原理在真实世界中，谁用出了名堂？最高决策者怎么用？执行层怎么用？今天做生意的怎么用？」然后到知识库中找对应笔记链过去。答案可能就在知识库已有的毛选/邓选/改革开放/黄奇帆等笔记中。

**典型案例**：ZN-06 孙子兵法精要的跨模块链接就是按这个三域模型构建的——毛的战略层、刘邓的战术层、华为拼多多特斯拉的商业层。

## 特殊场景：用户投喂教材/整书（MinerU转换的Markdown或直接投喂的原文Markdown）

**场景**：用户将一本教材/书籍通过 MinerU（或类似工具）转为 Markdown 格式丢过来，通常是按 MinerU 拆分规则分成上下册（`书名_上.md` + `书名_下.md`）。

### 核心规则：是否存原文由用户主动指定

用户约定：**「需要收录原文，我会特意说明。没有特意说明的，都做结构化分析框架笔记。」**

| 用户说了「收录原文」 | 用户没特意说 |
|:--------------------|:-----------|
| ✅ 合并原文→古籍全文/ + 更新分析框架笔记链接 | ✅ 只创建/更新分析框架笔记（六韬史鉴/），不存原文 |

### 执行流程

1. **检查拆分状态** — 确认收到的是完整教材还是单卷。如果收到多文件（上/下），用 `head`/`tail` 检查衔接处是否连续，确认无遗漏。

2. **确认用户意图** — 用户是否说了「收录原文」？
   - **是** → 执行流程3-6（原文入库 + 框架笔记更新）
   - **否** → 跳过流程3-5，仅执行流程6（只更新框架笔记，版本号+source注记）

3. **清洗原文** — 合并前后需清洗 MinerU 杂质：
   - 移除 `![image](https://cdn-mineru...)` 图片链接（grep 确认清零）
   - 压缩多余空行（连续3+ → 2）
   - 去掉行首尾空白
   - 添加文件头（书名/作者/译者/版本/ISBN）

4. **合并原文并入库** — 用 `cat` 将多文件合并为单文件，写入 `古籍全文/`：
   ```bash
   cat 书名_上.md 书名_下.md > "古籍全文/书名.md"
   wc -l -c 确认完整性
   grep -c 'cdn-mineru' 确认已清零
   ```

5. **更新分析框架笔记** — 在对应模块更新或创建框架笔记：
   - `source` 字段加入原文链接：`[[古籍全文/书名]]`
   - `updated: YYYY-MM-DD`
   - `version` 递增（如 v2.5→v3.0）
   - 若没有框架笔记，按标准格式创建（章节结构→核心模型→跨模块关联）

6. **建立双向链接** — source 字段的 wikilink 自动实现双向图谱。

### ⚠️ 关键坑

| 坑 | 说明 | 对策 |
|:---|:-----|:-----|
| **上下册衔接错位** | MinerU 拆分时可能按文件名而非内容逻辑 | 合并前用 `head`/`tail` 检查首尾衔接 |
| **框架笔记首选项** | 如果已有框架笔记（如 DS-14），是更新还是新建 | **优先更新已有笔记**，在 source 中追加新原文链接，不要开新笔记分裂同类内容 |
| **MinerU 图片链接** | MinerU 转出的 Markdown 含大量 CDN 图链接 `cdn-mineru.openxlab.org.cn`（单本教材常有150+个） | **必须移除**。用 `re.sub` 或 Python 脚本批量清理，grep 确认清零后再入库 |
| **大文件合并** | 合并后动辄 700KB~1MB，`read_file` 无法一次性读取 | 用 `terminal cat >` 合并，`wc -l -c` 验证，不经过 Python read/write |
| **用户没说要存原文** | 默认场景 | 只更新分析框架笔记，`source` 注明「用户已提供原文文件，未要求入库」 |

## 特殊场景：跨模块对话沉淀——「大脑撸铁」训练模式（Cross-module Dialogue → Bridge Note）

**场景**：用户问了一个「X 在 Y 框架下怎么看」的问题，X 和 Y 分别属于知识库的不同模块（如克鲁伊夫属智脑、易经属易哲）。回答本身就是一个跨模块合成物，值得永久沉淀。用户将此过程称为**「大脑撸铁」**——每次跨模块对话就是一组认知硬拉。

### 完整训练闭环

```
用户问跨模块问题（负重）
  → 我搜两边笔记（调动力量）
  → 我回答（完成动作）
  → 用户追问深层关系（加片）
  → 再搜+再生成（再拉一组）
  → 落回决断之桥（肌肉修复 = 入库）
```

用户说「比读书更有效」——因为多出了「质疑、检索、否定、重组」四个环节。每完成一次闭环，知识库就多一条神经突触。

### 触发条件
- 用户说「从我的知识库看，XX 和 YY 有什么关系？」
- 用户说「XX 用 YY 原理怎么解读？」
- 用户说「XX 和 YY 之间是怎样的关系」或「XX 是否继承了 YY」
- 用户明确要求你将两个现存笔记的内容串联分析
- 你的回答需要同时读取两个不同模块的笔记才能产出

### 沉淀判定
不是所有跨模块问答都值得沉淀。判定标准：
- **结构化对称**：两个知识体系有可映射的结构（如克鲁伊夫角色互换→泰卦天地交泰）
- **关系型分析**：两个人物/思想有可追溯的传承或对比关系（如毛邓同源异流）
- **用户主动要求「沉淀」**：用户说「沉淀下来」「写进库」「太好了」是最高优先级信号
- ❌ 不沉淀：纯粹的事实问答（如「毛泽东哪年出生」）、一次性修辞、无结构性的类比

### 执行步骤

1. **同时读取两边笔记** — 用 `search_files` 或 `read_file` 获取 X 和 Y 的核心内容
2. **找出底层共通结构** — 不要停留在表面比喻，找结构性的对称：
   - 克鲁伊夫「角色互换」= 泰卦「天地交泰」
   - 段永平「做对的事」= 克鲁伊夫「踢简单的足球」
   - 核心原则：找**同一张底牌的不同花色**，而不是编造类比
3. **产出结构化对应表** — 用表格呈现映射关系（X概念 ↔ Y概念 ↔ 共同底层逻辑）
4. **写入决断之桥** — 新笔记放在 `决断之桥/` 目录，按序号递增：
   - 文件名：`NN-X与Y的桥接.md`（如 `05-卦象与体育哲学.md`）
   - frontmatter 中 tags 包含两边模块名
   - 正文分三部分：① 跨模块对话的起点 ② 核心映射表格 ③ 逐条详解
5. **更新桥索引** — 在 `决断之桥/00-决断关系总论.md` 的子桥体系列表追加一行
6. **更新六韬总纲** — 决断之桥的 ASCII 树行 + 模块状态表

### 判定标准

| 适合沉淀为桥 | 不适合 | 
|------------|--------|
| 两个知识体系有结构性的对称 | 只是在找关键词匹配 |
| 映射后能发现「同一张底牌」 | 牵强附会的跨域类比 |
| 对未来的思考有启发价值 | 一次性的修辞技巧 |

### 🔄 桥笔记升级模式（Bridge Note Upgrade）

**场景**：已有桥笔记（如`07-说话的哲学`），用户提供了新的关联素材（如《百年孤独》），需要在桥笔记中**增加一面新的镜鉴**，而不是新建笔记。

> 这也是深度模式的一种——对应「深度模式」章节的 `模式 1.5`。触发时回退到该章参考版本语义和 v1→v2→v3 升级路径。

与创建新桥笔记的区别：

| 维度 | 新建桥笔记 | 升级桥笔记 |
|:----|:---------|:---------|
| 触发 | 用户问跨模块问题 → 新建 | 新的关联素材出现 → 追加 |
| 文件 | 创建 `NN-新文件名.md` | 修改已有文件 |
| 核心操作 | 从头写框架+映射表 | 插入新节+扩展现有表格 |
| 涉及范围 | 一条笔记 | 1条桥笔记 + N条源笔记（加反链） |

**升级七步法（已验证：07-说话的哲学 三重→四重镜鉴）：**

1. **插入新镜鉴节** — 在现有镜鉴序列的末尾插入新节（如百年孤独作为第四面镜鉴）。注意：按内容逻辑顺序插入（不要插在最后），更新节号。
2. **扩展所有对照表** — 每张表格增加一行/一列。关键表：①结构对称表(书的命题×各家回应→加新列) ②差异化定位表(加一行新定位)
3. **更新对普通人的启示** — 新增一条原则 + 一句话速记卡新增一行
4. **新增金句区** — 在新素材的 金句集 下追加
5. **更新知识库定位图** — ASCII图加一行新链接
6. **更新认知增量** — 改写"进化路径"描述，扩展坐标轴
7. **注入反向链接** — 新镜鉴的源笔记添加关联阅读指向桥笔记

**⚠️ 坑**：因为是在现有笔记上做修改，所有 patch 操作都要注意上下文不唯一（"四"和"三"的区别）。建议在 patch 前用 `read_file` 读全文确认当前内容，防止插错位置。

## Vault架构维护

> **关联参考**：`references/vault-restructuring.md` — 大规模知识库架构重构六步法（命名体系升级 / 编号冲突修复 / 批量链接刷新）

来自花叔橙皮书的经验：一个健康的vault需要有一份"宪法"来统一笔记格式。当前用户vault（103篇笔记）的 frontmatter 格式已统一，但日常维护仍需注意。

### 统一标准文档

vault 内 `_标准/FRONTMATTER规范.md` 定义了全局 frontmatter 规范：

- **必选字段**：`title` + `tags`（YAML列表格式）+ `created`
- **推荐字段**：`aliases`（列表）+ `type` + `status`（状态标签）
- **严禁使用**：`tags: [a,b]` 内联格式或 `tags: #a #b` 哈希格式
- **tags 一律用 YAML 列表格式**：`tags:\n  - 模块名\n  - 关键词`

更新到这个标准后的 vault 状态验证：
```
python3 -c "
import os, yaml, re
from pathlib import Path
issues = []
for root, _, files in os.walk(VAULT):
    for f in files:
        if not f.endswith('.md'): continue
        m = re.match(r'^---\n(.*?)---\n', Path(root,f).read_text('utf-8'), re.DOTALL)
        if not m: continue
        fm = yaml.safe_load(m.group(1))
        if not fm: continue
        if 'tags' in fm and isinstance(fm['tags'], str):
            issues.append(f'{f}: inline tags')
        if 'title' not in fm:
            issues.append(f'{f}: no title')
print(f'Issues: {len(issues)}') if issues else print('All clean')
"
```

### 01-信息流/ 收件箱净化规范

收件箱只应包含两类文件：
1. `00-喂料模板.md` — 模板
2. `status: raw` 或 `status: processed` 的**素材笔记**

非素材文件（索引总览、MOC 地图等）必须移出：
- `01-知识图谱总览.md` → `MOC地图/知识图谱总览.md`

检测：`ls vault/01-信息流/`，如果看到 `00-` 以外的序号前缀文件（如 `01-...`），很可能是非素材索引，需要移动。

### 决断之桥索引校验

`决断之桥/00-决断关系总论.md` 的「子桥体系」列表容易与实际文件名脱节——早期版本列出了 `01-感知之桥`、`05-升维之桥` 等文件，但实际文件名为 `01-卦象与战略决策`、`04-灰度与易理`。每次在决断之桥新增笔记后，顺便检查该列表与实际 `ls 决断之桥/` 的结果是否一致，如有漂移就一次性修正（包括名称和连接线描述）。

### 用户执行偏好：少干预

每次执行喂料流程时，注意以下原则（来自用户明确表达过）：

先自问 **韬定律元操作三问**（源自 00-六韬总纲 核心理念）：\n1. **做对的事 > 做更多的事** — 先停掉不必要的事，再判断新素材值不值得喂\n2. **改变连接方式 > 加新节点** — 优先用链接/桥笔记串联已有笔记，而非开新模块\n3. **修三次还不好就该换维了** — 同一个问题反复出现，说明当前方法论的维度已到极限\n4. **断舍离：半成品果断舍弃。被功能吸引不算拥有，果断删比留着占地方更有价值。** — 不是所有有用的东西都要背着走，博采众长 ≠ 背负一切

然后执行以下具体原则：

| 原则 | 说明 |
|------|------|
| **不追问细节** | 用户说"就做XX"就默认全面画像+标准流程推进 |
| **自决策优先** | 模块归属、v2判定、建链逻辑自己判断，做完了报告 |
| **只需必要时才问** | 只有需要密钥/密码时问一次，其余全自主 |
| **不要发起确认话题** | 用户说"解决""处理""看看这个"就去解决，**不要输出方案征集意见**。自主判断+直接执行，执行完呈报结果。用户说"你仔细想办法"就是让你直接想、直接做，不是让你回来确认方向 |
| **呈报结果不放过程** | 交付时给结论+数据，不给怎么做的步骤说明 |
定期巡检。详见 `references/vault-maintenance.md`。

### 扫描可靠性检测

**扫描方法**：使用 `execute_code` 读取完整文件内容，用 Python 的 YAML 前端解析或全量字符串搜索 `status: raw` / `status: processing`，而非 `head -N` + `grep` 局部扫描。

**为什么重要**：收件箱中有些注入笔记的 frontmatter 较长（10-15行），`status:` 字段可能在第8行或更后。`head -6` 或 `head -10` 扫描会漏掉这些文件，误判为未处理，浪费 token 去逐文件检测。**始终读完整文件再判断 status**。

**All-clean 情形**：如果所有素材均已 `status: processed`，则本次扫描无产出。对于 cron 任务，回应 `[SILENT]` 抑制空交付。不做任何操作。

#### 方案A：Tesseract（标准方案）
工作流：

#### 方案B：RapidOCR（当PaddleOCR/EasyOCR均不可用时）
**适用场景**：PaddlePaddle的CPU不兼容(SIGILL/AVX512缺失)、GitHub不通导致EasyOCR模型无法下载、且无法用Tesseract时。

**优势**：基于ONNX Runtime，纯Python包（pip install rapidocr-onnxruntime），模型自动下载，无需C++编译。国内源可通。

**安装**：
```bash
pip install rapidocr-onnxruntime
```

**使用示例**（处理大图的分块OCR，防止OOM）：
```python
from rapidocr_onnxruntime import RapidOCR

engine = RapidOCR()
result = engine("image.jpg")
# result is a tuple: (detections_list, timing_info)
detections = result[0]  # list of [bbox, text, confidence]
if detections:
    for item in detections:
        text, conf = item[1], item[2]
        if conf >= 0.5:
            print(text)
```

**⚠️ 关键陷阱**：`RapidOCR()` 的返回值是 **元组** `(detections, timings)`，不是扁平列表。
- ✅ 正确：`result[0]` 获取检测结果
- ❌ 错误：直接迭代 `result`（会拿到时间数据而非文本）

**大图分块处理模式**（防OOM）：
```python
engine = RapidOCR()
all_text = []
for i in range(num_chunks):
    chunk_result = engine(f"chunk_{i}.jpg")
    detections = chunk_result[0]
    if detections:
        for item in detections:
            all_text.append(item[1])
    # engine实例可复用，不会累积内存
full = "\n".join(all_text)
```

**分块策略**：对超长截图（如929×11822像素），用 PIL/Pillow 切成每块约2000px高的分块再逐个OCR。
```python
from PIL import Image
img = Image.open("screenshot.jpg")
w, h = img.size
num_chunks = 6
chunk_h = h // num_chunks
for i in range(num_chunks):
    y1, y2 = i * chunk_h, (i+1) * chunk_h if i < num_chunks-1 else h
    chunk = img.crop((0, y1, w, y2))
    chunk.save(f"chunk_{i}.jpg", quality=90)
```

#### 方案共通：元数据提取兜底

如果所有OCR方案都无法使用，则：
- **先提取可用元数据**：很多扫描版PDF的水印/页脚中仍有少量可读字符（如"三个皮匠报告站 文档Id:1191225"等）。用这些碎片——来源网站URL、文档ID、碎片关键词——帮助用户定位文字版。
- 标记 `status: raw` 并告知用户需要文字版，附上提取到的元数据

- UTC+8 当天 00:00:00 的**毫秒级时间戳**

### 同步规则（严格执行）

1. **每次知识库操作后必须同步三表** — 不需要用户提醒或督促
2. **日期不留白** — 即使当天无任何操作，也要在每日记录和搭建日志中各留一条空白记录（仅日期+农历+干支+节气，内容写"今日无操作"）
3. **写入前先查重** — 用 `GET /records?page_size=50` 检查当天是否有记录，避免重复写入
4. **API字段名匹配** — 创建记录时使用 **字段显示名**（如"操作内容""时间""领域"），不是字段ID（`fldXXX`）。SingleSelect 字段传 **选项名称字符串**（如"知识库基建""✅已完成"），不是选项ID
5. **收藏随想录只在有新增外部素材时写入** — 外部素材指从微信文章/网页/PDF/书籍/聊天记录等来源收集的内容。内部生成的知识库笔记不属于收藏范畴

### 保底机制：每日 cron job

设立一个 no_agent 模式的每日 cron job，在每晚 22:00 执行独立的 Python 脚本：

```bash
# cron job 配置
name: 每日保底记录
script: daily_blank_record.py  # 放在 ~/.hermes/scripts/
no_agent: true
schedule: "0 22 * * *"
repeat: forever
```

脚本逻辑：
1. 获取飞书 tenant_access_token
2. 查询「每日记录」表当天是否有记录

4. 查询「搭建日志」表当天是否有记录
5. 无记录 → 写入"今日无知识库操作"

⚠️ **凭证处理**：脚本需要硬编码 APP_ID / APP_SECRET（因为 cron 会话不一定继承环境变量），但允许被环境变量覆盖：

```python
    resp = requests.get(f"{base_url}/tables/{TABLE_ID}/records?page_size=50", headers=headers)
    items = resp.json()["data"]["items"]
    has = any(r["fields"].get("公历时间") == today_ms for r in items)

    if not has:
        # 计算农历信息
        lunar_str, gan_zhi, huangli, jieqi = get_lunar_info()
        # 写入
        requests.post(f"{base_url}/tables/{TABLE_ID}/records", headers=headers, json={
            "fields": {
                "公历时间": today_ms,
                "农历": lunar_str,
                "节气": jieqi,
                "干支": gan_zhi,
                "黄历宜忌": huangli,
                "记录内容": "今日无操作",
                "类型": "💼 工作"
            }
        })
```

### 每日记录追加更新（同一日多轮操作）

当同一天内有多轮知识库操作时（如上午入库马哲、晚上入库芒格），**不允许创建重复的日期记录**。应更新已有记录的 `记录内容` 字段，追加新操作摘要：

```python
# 先查当天已有记录
resp = requests.get(f"{base}/tables/{daily_table}/records?page_size=50", headers=headers)
items = resp.json()["data"]["items"]
today_records = [r for r in items if r["fields"].get("公历时间") == today_ms]

if today_records:
    # 已有记录→追加内容（用换行分隔）
    rec_id = today_records[0]["record_id"]
    old_content = today_records[0]["fields"].get("记录内容", "")
    if "新操作关键词" not in old_content:
        new_content = old_content + "\\n新操作描述"
        requests.put(f"{base}/tables/{daily_table}/records/{rec_id}",
            headers=headers, json={"fields": {"记录内容": new_content}})
```

同样适用于搭建日志：同一天的多笔操作各自新建记录（每笔一条），先用 GET 查重避免重复写入同一操作内容。**直觉判断法**：操作内容相同就是重复、操作内容不同就是新记录。

### ⚠️ 字段选项 PUT 陷阱（实测踩坑）

当需要**修改字段的 SingleSelect 选项**（去重、重命名、删除冗余选项）时，千万不要直接用 PUT 替换整个 options 数组。准确操作流程：

```
❌ 错误做法：PUT 字段选项 → 所有记录的该字段值全部清空
✅ 正确流程：
   Step 1: 查出所有使用"脏选项"的记录（GET records）
   Step 2: 用字段 NAME（不是内部ID）迁移记录到正确选项（PUT record）
   Step 3: 再用 PUT 更新字段选项列表，移除脏选项
```

**为什么 PUT 字段选项会清空记录值**：PUT 会重建所有选项的内部 ID（option_id），即使选项名称不变，新 ID 也与原有记录存储的旧 ID 不匹配，导致 API 返回空字符串。

**根本原因**：`PATCH /fields/{field_id}` 返回 404（飞书 v1 API 不支持 PATCH 单字段），只能用 PUT 替代。而 PUT 是全量替换，没有部分更新语义。

**关键代码**（PUT 字段选项的正确写法）：
```python
# Step 1+2: 先迁移记录（用选项 NAME）
requests.put(f"{base}/tables/{table_id}/records/{record_id}", json={
    "fields": {"状态": "✅ 已完成"}  # 传选项显示名，不是内部id
})

# Step 3: 再更新字段定义
requests.put(f"{base}/tables/{table_id}/fields/{field_id}", json={
    "field_name": "状态",
    "type": 3,  # SingleSelect
    "ui_type": "SingleSelect",
    "property": {
        "options": [{"color": 0, "name": "✅ 已完成"}, ...]  # 不包含脏选项
    }
})
```

**选项去重/美化时的注意事项**：
- 选项名头尾空格会被视为不同选项（如 `"✅已完成"` vs `"✅ 已完成"` 在用户看来一样，但对 API 是不同选项）
- 选项名中的 emoji 前缀要统一风格（该用户有"审美追求"偏好——用 emoji 前保持视觉一致性，如 `其他→🗂 其他`, `已入库→✅ 已入库`）

## 深度模式（Deep Mode）

> **统一入口**：深度模式在本文档中共有三个触发点——
> 1. **主流程步骤14**（后方「判断是否走深度模式」）→ 决定走 v1 还是深度模式
> 2. **桥笔记升级模式**（本章「模式 1.5」）→ 已有桥笔记追加新镜鉴
> 3. **大型素材处理**（本章「模式 4」）→ 整书并行拆解，含自身 Phase 1-5 子工作流

三者共享同一套**版本语义约定**：

| 版本 | 含义 |
|:----:|------|
| v1.0 | 首次入库，标准提炼 |
| v2.0 | 深研版：多源交叉验证+跨思想家对比+普通人启示 |
| v3.0+ | 全集原文血肉化：每论点有第一手原文支撑 |

深度模式分为四种场景，按深度递增排列：

```
素材类型 →
├─ 人物思想/管理哲学/v2 覆盖     → 1. 深度挖掘模式（v2）
├─ [桥笔记新关联素材]           → 1.5 桥笔记升级模式（追加镜鉴）
├─ 已有v2笔记 + 全集原文出现     → 2. 后期原文注入（v3）
├─ 同作者两本独立著作同时出现    → 3. 双书交叉注入
└─ 整书 200页+ / 需要并行拆解   → 4. 大型素材处理
```

### 1. 深度挖掘模式（v2版：人物/思想深研）

**走 v2 的条件**：思想/哲学/方法论类素材（人物思想/管理哲学/传记）→ 直接走 v2 深度版。纯信息类素材（报告/数据/行业分析）→ v1 标准入库。历史/传记（以人物为核心的叙事）→ 史鉴模块走 v2。法律/伦理/哲学类 → 太极双螺旋模块走 v2。

- **「对普通人的启示」是必选项而非可选项** ⭐
- **跨思想家横向对比是必选项** — 每个深研版必须有 vs 知识库中其他思想家的对比（最少3个对照维度，用表格呈现）。这建立思想家之间的图谱关系，防止孤立节点。
- v2 规格：500-1000行，以下章节结构（已验证5次，与经用户认可）：
  1. **人物定位** — 为什么这个人值得学（1-2节）
  2. **思想体系** — 核心支柱展开（4-6根支柱，每支柱含：理论+案例+警示）
  3. **经典案例深度拆解** — 每个案例单独成节，含背景→逻辑链→结果→启示
  4. **vs其他思想家的横向对比** — 最少3个对照维度，定位每个人在知识库中的角色
  5. **对普通人的启示** — 这是用户最认可的部分（用表格/清单展现，可操作）
  6. **金句集** — 思想家的精华语录（15-25条，带背景说明）
- **在六韬知识库中的定位** — ASCII图展示与其他模块/笔记的链接关系。注意知识库是一座活的树：双螺旋=根系（底层思想范式），史鉴=主干（经验视角），智脑=枝干左（理性方法论），易哲=枝干右（感性智慧），决断之桥=枝干交汇（理觉合一），实战案例=树冠（验证沉淀）。根系会破土成枝（双螺旋模块直接从根系长出）
- 模块定位规则：
  - 商业哲学/管理思想 → 六韬智脑（段永平/曾鸣/黄奇帆）
  - 历史/传记 → 六韬史鉴（邓小平时代）
  - **法律/伦理/哲学 → 太极双螺旋**（罗翔——作为双螺旋的哲学底座）
- v2 直接在原文件上覆盖（保持文件名和链接不变），更新 `version: v2.0`

### 2. 后期原文注入（v3版：原文血肉化）

**触发条件**：已有 v2.0 深研笔记，全集/原版全文原文出现。

**产出**：笔记升级到 v3.0——每论点有第一手原文支撑，将原文引文血肉化到每个框架支柱中。

**参考**：详见 `references/deep-dive-template.md`。

### 3. 双书交叉注入

**触发条件**：同一位作者的两本独立著作同时进入处理（或前后出现）。

**产出**：金句翻倍 + 书间对比表——提炼两本书共同覆盖的核心模型，标注观点演进/补充/修正。

**参考**：详见 `references/deep-dive-template.md`。

### 4. 大型素材处理（整书并行拆解）

当素材为整书（200+页）或同一作者的姊妹篇时，走深度模式而非标准提炼。

**场景**：用户发送/分享一整本书（300页+的PDF），需要从原始资料到深度知识库条目的全链路处理。这不同于文章/报告级素材——体量大、结构复杂、需要多角度并行分析。

#### 与标准素材处理的区别

| 维度 | 文章/报告（~20页） | 整书（200页+） |
|:----|:-----------------|:--------------|
| PDF提取策略 | 全量提取即可 | 先扫目录+结构分析，再全量提取 |
| 内容组织 | 单次阅读→提炼 | 按章节切分→委托子代理并行分析 |
| 跨模块潜力 | 单个模块 | 可产生多条跨模块链路 |
| 总纲更新 | 单个场景入口 | 模块地图+使用场景+跨模块关联表全更新 |
| 反链注入 | 2-3条 | 5条+反向链接注入 |

### 执行步骤

#### Phase 1: PDF 元数据检查与结构扫描

```python
import fitz
doc = fitz.open("book.pdf")
print("页数:", doc.page_count)
# 读目录页（通常p2）确认章节结构
toc_text = doc[1].get_text().strip()
# 扫描每页文本量，定位无文本/图片页
for i in range(doc.page_count):
    text = doc[i].get_text().strip()
    if len(text) < 50:
        print("第%d页无文本或极少文本（%d字符, %d张图）" % (i+1, len(text), len(doc[i].get_images())))
# 判断是否为纯文本PDF（301+/305有文本 = 好质量）
print("总字符数:", sum(len(doc[i].get_text()) for i in range(doc.page_count)))
```

**判断标准**：如果 95%+ 的页有文本（>50字符），直接走 PyMuPDF 提取全量。如果大量为扫描页，转 RapidOCR 或告知用户。

#### Phase 2: 章节切分与子代理并行委托

根据目录页和关键词定位，将全书按章节切成逻辑块，委托3个子代理并行分析：

| 子代理 | 处理内容 | 典型产出 |
|:------|:---------|:---------|
| Agent A | 核心方法论/思维模型章节 | 模型清单+决策框架+原文定义 |
| Agent B | 最具特色的专题章节（如25种心理倾向） | 完整逐条提取+跨知识库对比分析 |
| Agent C | 生平/传记/语录章节 | 生平关键时刻+核心哲学+金句集 |

**委托指令模板：**
```python
delegate_task(tasks=[
    {
        "goal": "从X书中提取核心思维模型和决策方法论",
        "context": "以下是第Y-Z章(%d字符)的原文全文。提取要求：模型清单+决策框架+定义原文引用",
        "toolsets": ["terminal", "file"]
    },
    # ... 另外两个task类似
])
```

**关键参数**：每个子代理的 context 中传入该章节的完整原文（非摘要），让子代理基于原文而非记忆工作。子代理产出的文件路径要明确写出，供父代理合成用。

#### Phase 3: 融合创作（Synthesis）

子代理完成后，父代理做三层融合：

1. **读取子代理产出**：子代理的 summary 字段内容有限（只含整体摘要），真正的详细产出写在各子代理创建的独立文件中（如 `/home/hermes/tmp/xxx_structured_extraction.md`）。父代理必须用 `read_file` 读取这些文件获取完整内容后再合成。不要仅凭 summary 合成——summary 提供索引，文件提供细节。
2. **结构融合**：按「人物定位→核心哲学→核心模型→特色专题→金句→跨模块对话→定位」的组织框架，将三段产出拼接
2. **跨模块对话**：主动建立与知识库已有条目的关联（至少5条双向链路），包括：
   - 与主干笔记（史鉴）的思想对比——同哲学流派/历史源流
   - 与枝干笔记（智脑）的方法论对接——决策工作流/灰度治理/信号博弈
   - 与根系（双螺旋）的认知框架联系——理性防御/感性体验/三大开关
   - 注意树形方向：主干的笔记向上链枝干，枝干的笔记向下链主干，根系破土的新枝干直接链根系
3. **总纲三处更新**（见下）

#### Phase 4: 总纲三处更新 + 反向链接注入

创建笔记后，更新三处总纲：
- `00-史鉴总纲.md`（或对应模块总纲）：模块地图 + 跨模块关联表
- `00-六韬总纲.md`：树结构行 + 模块计数 + 使用场景速查

反向链接注入（最容易遗漏）：
- 在相关现有笔记的「关联阅读」区加一行
- 在对比段落插入对照引用
- 用 `search_files(pattern="新笔记名")` 验证至少2条外部链接

#### Phase 5: 飞书三表同步

```
搭建日志 → 新增1条（描述完整操作）
每日记录 → 追加当日记录（若已有则更新内容）
收藏随想录 → 仅在有外部素材时写入
```

### 已知陷阱

| 陷阱 | 现象 | 对策 |
|:----|:-----|:-----|
| **子代理输出过大** | 某子代理产出31KB+，超过父代理单次读取能力 | 子代理输出保存为文件，父代理只读取关键摘要部分合成 |
| **章节边界误判** | 目录页中的关键词与实际内容位置偏差 | 以实际内容的首次出现位置为准，不依赖目录页页码 |
| **跨模块对比太泛** | 生硬类比（'芒格和孔子都很智慧'） | 必须有结构性对称（如人性论的系统观）才能入库，否则不写 |
| **总纲格式陷阱** | 含 `[[链接\\\\|别名]]` 的行在 patch 时产生 `|||` | 逐行 patch，读确认后再下一行 |
| **代码块展平陷阱** | patch 的 new_string 含多行代码块（```...```）时，内容被展平为 `\\n` 转义字符串，不会渲染为实际换行 | 先用 patch 替换旧代码块，再用 `read_file` 检查结果；如果被展平，写第二个 patch 用正确的多行代码块替换展平字符串。**不要**在 new_string 中使用代码块起始结束符 ```— 改用独立的新帖补丁分两步走 |

### 精要+精读多文件拆分

当精读对象是**多卷/大部头经典**（如《毛泽东选集》全四卷），单一精要笔记不足以承载全文深度时，采用「精要框架 + 多卷精读」的拆分结构：

| 层级 | 文件 | 功能 | 典型规模 |
|:----|:-----|:-----|:--------|
| 精要 | DS-05-毛泽东选集精要.md | 方法论框架、六韬关联映射 | 143行 / 8KB |
| 精读V1 | DS-05-V1-毛泽东选集第一卷精读.md | 7篇核心文章逐篇精读+原文金句 | 263行 / 21KB |
| 精读V2 | DS-05-V2-毛泽东选集第二卷精读.md | 论持久战·新民主主义论等 | ~360行 / 18KB |
| 精读V3 | DS-05-V3-毛泽东选集第三四卷精读.md | 改造学习·人民民主专政等 | ~400行 / 21KB |

**适用条件**：全文 > 50万字 或 核心文章 > 15篇。
**命名规范**：精要笔记保持原名，精读笔记用 `AA-NN-V{1..N}-标题.md`。
**总纲更新**：精要笔记的「关联阅读」加指针行，总纲下展开子条目。

3. **Obsidian 表格格式陷阱** — 含 `[[链接\\|别名]]` 的表格行在 patch 时易出两种错：
   - **行首误成 `|||`（三竖线）** —— `patch` 工具在匹配以 `|` 开头的旧行时，会在新行前自动加上一个 `|` 前缀，导致 `|` + `||` = `|||`。这是工具行为，非语法错误。**预防**：new_string 不写开头的 `|`，纯写内容让工具补；或 old_string 不带前导 `|` 用整行内容匹配。
   - **链接内 `\\|` 误成 `\\\\|`（双反斜杠变四）** —— 变量替换导致转义加倍
   - **还可能出现 `|}` 孤行**（patch 意外插入了表格终止符）
   - **诊断**：`read_file` 查看表格区域，正常行以 `|` 开头，别名用 `\\|`。异常行看有无 `|||`、`|}` 或 `\\\\|`
   - **修复**：用 `patch` 只替换坏行，new_string 手动写 clean 版本（单 `|` 开头 + 单 `\\` 别名）。**不要**一次写整个表格——逐行 patch 风险更低、更可控。
   ```python
   # 修复 ||| → |
   patch(path=note.md,
       old_string="||| [[笔记名\\\\\\\\|别名]] | 内容 |",
       new_string="| [[笔记名\\\\|别名]] | 内容 |",
   )
   # 修复 |} 孤行（patch意外插入的表格终止符）
   patch(path=note.md,
       old_string="|}\\\n### 下一节标题",
       new_string="### 下一节标题",
   )
   ```
   - **注意**：修复后必须 `read_file` 确认整行都恢复为 `|` 开头，而不是 `|||` 或 `|}`。Obsidian 表格渲染失败时排查顺序：`|||` → `|}` → 别名内 `\\` 变 `\\\\`。
