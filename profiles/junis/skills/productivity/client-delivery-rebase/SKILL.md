# 客户端交付包适配管线（Client Delivery Rebase）

## 核心理念

内部工具和提案默认带着我们的品牌烙印（团队代号、内部命名、技术栈偏好）。给客户交付前，需要一次"rebase"操作——剥离内部痕迹，适配客户环境，让客户拿到手就像自己的东西。

> **原则：客户拿到的应该是"他们的方案"，不是"我们的方案改了个名"。**

---

## 标准 Rebase 三步

### 第一步：品牌去痕

扫描所有交付文件，替换以下内容：

| 找什么 | 替换为 | 说明 |
|:-------|:-------|:-----|
| 内部团队代号（如"六韬"） | `[客户代号]` | 全局替换，交付时客户填代号后一键替换 |
| 内部Bot团队名（如"六韬Bot团队"） | `[客户代号]Bot团队` | 如有自定义团队名可进一步定制 |
| 内部知识库名（如"六韬·认知操作系统"） | `[客户代号]·认知操作系统` | 保持"认知操作系统"名称不变，前缀替换 |
| 作者署名（如"作者：六韬 / FDE"） | `作者：[客户代号] / FDE` | FDE框架名保留，团队名替换 |
| 模块前缀（如"六韬智脑/六韬史鉴/六韬易哲"） | 替换为通用术语或删去前缀 | 内部模块命名对客户无意义 |
| 框架版本脚标中的团队信息 | `[客户代号]` | 视觉图、架构图等脚标处 |

**检查命令：** `grep -r "六韬\|内部代号" 交付包目录` — 必须零命中才能交付。

### 第二步：管道适配

将内部默认输入管道切换为客户可触达的工具：

| 内部默认 | 客户端推荐替代 |
|:---------|:-------------|
| Hermes Agent / 飞书机器人 | WorkBuddy + 微信（零配置） |
| 飞书/钉钉事件订阅 | WorkBuddy 自动化 + 微信转发 |
| @Bot 指令 | 微信消息 → WorkBuddy 自动捕获 |
| 飞书知识库 | WorkBuddy 知识库 + 微信收藏 |
| Obsidian 本地优先 | WorkBuddy 共享 / 飞书（按客户选择） |

**改动范围**（以FDE主提案为例）：
- §4.x 使用方式：飞书/钉钉的操作说明 → 微信/WorkBuddy的操作说明
- §6.x 技术选型：飞书SDK/事件订阅 → WorkBuddy API + 微信生态
- §6.x 架构图：接口调用箭头 → 自动流式采集
- §6.x 案例：案例中的平台名替换
- §10.x 竞品对比：确保微信作为优势被突显
- L1输入层表格：工具示例替换
- 所有附录/代码中的主方案标注

### 第三步：客户自定义

在每个交付包中加入客户信息占位区：

```markdown
## 📋 客户信息

> 请在交付前填写以下信息，所有 `[客户代号]` 占位符将统一替换。

```
客户名称：_________________________________
客户代号（用于知识库命名）：_________________
首选输入工具：☐ WorkBuddy + 微信（推荐） ☐ 飞书 ☐ 钉钉 ☐ 其他：_______
交付日期：_________________________________
```
```

**交付流程：**
1. 填完客户信息表
2. 全文替换 `[客户代号]` → 实际代号
3. 删除客户信息占位区本身（不让客户看到自己的占位符）
4. 可选：按客户首选工具调整备选方案标记

## 备选组件处理

内部工具链中有些组件是面向特定平台的（如飞书bot、Hermes MCP）。**在客户端交付中应直接删除，不保留为「备选方案」。**

| ✅ 正确做法 | ❌ 错误做法 |
|:-----------|:-----------|
| 删除平台特定代码，保持交付包干净 | 保留并标注"备选方案" |
| 客户如需该平台能力，单独作为加购项提供 | 让客户困惑"我到底该不该用这个文件" |
| 在README FAQ中提一句"可联系获取XX对接方案" | 在交付包中保留完整的平台特定代码 |

**原则：** 客户的交付包应该只有一条明确的路径可以跑起来。多条路径=多条困惑。

### 代码依赖剥离

当交付包中包含内部工具代码（如 rest_server.py）时，需要剥离内部框架依赖：

| 需替换的 | 替换方案 | 原因 |
|:---------|:---------|:-----|
| 内部类引用（如 `HermesMemoryVectorAdapter`） | BM25-only 模式，或抽象接口+优雅降级 | 客户没有内部框架 |
| 内部路径（如 `~/.hermes/hybrid-memory/`） | `~/.baodating/memory/` 或环境变量 | 避免路径依赖 |
| 内部环境变量（如 `HERMES_HOME`） | 通用环境变量（如 `HYBRID_MEMORY_DB`） | 品牌中立 |
| 向量检索引擎（如向量Embedding） | BM25-only 降级，文档注明升级路径 | 低成本开箱即用 |

**模式：** 用一个 `try/except ImportError` 包裹内部导入，优雅降级到 BM25-only。这样客户无论有无内部组件都能运行。

## 创建客户端可部署组件

除了清理和适配，还需补充客户端可直接使用的技能包：

| 组件 | 说明 | 必须包含 |
|:----|:-----|:---------|
| **WB 主脚本** | WorkBuddy 自动化调用的 CLI 入口 | process/batch/confirm/pending/stats 子命令 |
| **配置文件模板** | 客户填入 API Key 即可用 | api_key, api_base_url, model, customer fields |
| **一键安装脚本** | python3 install.py 搞定所有配置 | 环境检查+依赖安装+目录创建+自动化配置指导 |
| **部署指南** | 三步部署说明，非技术人员也能看懂 | 前置条件→安装→配置→测试→常见问题 |

### WB Skill 脚本设计要点

```
baodating.py
├── process <text>    → 接收用户输入 → AI提取7字段 → 返回确认
├── confirm <reply>   → 1/改到XX/忽略 → 更新知识库
├── pending           → 查看待确认列表
├── batch             → 批量处理（-f 文件 / -t 多条文本）
└── stats             → 按分类的统计数据
```

- 支持 CLI 直接传参 + stdin 管道输入（适配 WorkBuddy 两种触发方式）
- 用 JSON 文件做状态管理（无需数据库，零配置）
- 提供 DeepSeek/Qwen/OpenAI 兼容接口（客户可选模型）

## 改动范围估算

| 交付包规模 | 文件数 | 改动行数 | 预估工时 |
|:----------|:------|:--------|:--------|
| 小型（1主提案+1-2附件） | 3-5 | 10-20 | 10-15min |
| 中型（1主提案+代码+Prompt+视觉） | 5-8 | 30-60 | 20-40min |
| 大型（多文档+多语言+多平台） | 10+ | 60-100+ | 40-60min |

## 实战案例：包打听 v1.1 客户端定制

> **时间**：2026-06-08
> **原始交付包**：FDE-第二大脑企业交付框架-完整提案.md + code/ + assets/ + prompts/
> **改动量**：3个主文件，~30处替换，4小时（含燕青前置吸收工作）

**具体改动的4个文件：**

| 文件 | 品牌去痕 | 管道适配 | 客户自定义 |
|:----|:--------|:--------|:----------|
| 主提案 | 六韬→[客户代号]（6处） | 飞书/钉钉→WorkBuddy+微信（10+处） | 首页新增客户信息框 |
| 架构文档 | 六韬→[客户代号]（2处） | Hermes→[客户代号]（3处） | — |
| System Prompt | — | 飞书妙记/文档→通用名称（4处） | — |
| 知识树视觉图 | 六韬→[客户代号]（1处） | 版本号v1.0→v1.1 | — |

**关键经验：**
- 先全局搜索，再逐处修改。用 `search_files` 统计所有命中位置再做计划
- pipeline重写时，确保案例中的平台名与正文一致（否则客户会困惑）
- **平台特定代码（如 feishu_bot.py）直接删除不留备选**——客户如有需要，做独立加购项
- 注释/附录中的引用也要改（容易遗漏）
- 代码中的内部框架依赖（Hermes类引用/路径/环境变量）必须剥离，优雅降级到轻量模式

## ⚠️ 常见陷阱

| 陷阱 | 现象 | 对策 |
|:----|:-----|:-----|
| **附录引用遗漏** | 附录E描述仍写"与Hermes现有架构关系" | 改主提案时同步检查附录表 |
| **版本号不一致** | 视觉图写v1.0但主提案写v1.1 | 交付前用grep确认版本号一致性 |
| **备选组件残留** | 平台特定代码留在交付包中，客户困惑是否要部署 | 直接删除，客户如需通过加购项提供 |
| **占位符残留** | `[客户代号]` 留在最终交付物中 | 交付前全局搜索 `[客户代号]` 确认已替换 |
| **代码依赖残留** | rest_server.py 引用内部Hermes类，客户无法运行 | import包在try块中优雅降级，去掉内部路径硬编码 |
