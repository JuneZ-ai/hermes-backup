# Grill Me — 动手前先被审问

> **核心理念：** 没人一开始就知道自己要什么。动手前先花 3-5 分钟把需求每个分支走一遍，能省下 80% 的返工时间。

## 使用方式

### 触发场景

以下场景**必须**加载本 skill：

1. 用户说「帮我做一个 X 功能」但只有一句话
2. 用户说「重构/修改 X」但不清楚具体要改什么
3. 用户说「开始搞」但没有需求文档
4. 涉及跨模块/有副作用的变更

以下场景**不需要**加载：

1. 已经有明确的 PRD/规格文档
2. 纯工具操作（查个 API、读个文件）
3. 极其简单的一步操作（改个文案、改个颜色）

### 审问流程

逐项提问，**一次只问一个问题**，等用户回答后再问下一个：

#### 第一轮：基础定位

1. **要做什么？** — 用一句话概括你要改的东西
2. **为什么做？** — 解决了什么问题 / 创造了什么价值
3. **谁在用？** — 最终用户是谁，他们的使用场景是什么

#### 第二轮：边界与约束

4. **成功标准** — 怎样算做好了？有验收标准 / 测试用例吗？
5. **不做清单** — 这个需求的「不 scope」有哪些？明确排除什么？
6. **现有制约** — 有什么必须兼容的东西？API 格式？数据库 schema？UI 规范？

#### 第三轮：风险与决策

7. **风险点** — 你觉得哪里最容易出问题？
8. **备选方案** — 考虑过其他做法吗？为什么选这个？
9. **优先级** — 这个需求多紧急？有 deadline 吗？

### 响应模板

审问结束后，输出一个「需求理解确认」：

```
## ✅ 需求理解确认

### 要做的
[一句话概括]

### 不做
- [排除项 1]
- [排除项 2]

### 验收标准
- [ ] [条件 1]
- [ ] [条件 2]

### 风险提醒
⚠️ [已知风险]

---
用户确认后 → 开始执行
用户要求改 → 回到对应轮次重新审问
```
