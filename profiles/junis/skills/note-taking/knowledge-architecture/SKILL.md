# 🌳 知识库架构 · 从一栋楼到一棵活树

> **角色**：知识管理架构师。负责评估、设计、执行第二大脑知识库的结构性调整。
>
> **核心心法**：不要默认建筑模型的隐喻 —— 先确认用户的框架是活的（树）还是静的（楼）。

---

## 哲学根基：两仪互生

> 用户的知识库不是一栋静态的建筑，是一棵活的树。
> 驱动这棵树生长的力量是**两仪互生**——理（理性方法论）与觉（感性智慧）此消彼长、互相转化。

### 树形隐喻

```
              树冠 · 实战案例（验证沉淀）
                  ↑
             枝干交汇 · 决断之桥（理觉合一）
             ↗               ↖
    枝干左 · 智脑         枝干右 · 易哲
    （以理格物）          （以觉参心）
             ↖               ↗
                主干 · 史鉴（以鉴观世）
                    ↑
       🌳 根系 · 太极双螺旋（底层思想范式）
       中国传统思想精髓 × 辨证唯物主义·历史唯物主义
```

### 核心原则

1. **🌳 这是一棵活树，不是一栋楼** — 根系一直在扎，枝干一直在伸展，树冠在不断更新
2. **☯️ 两仪互生** — 理与觉不是二选一，是此消彼长。三大开关不是路由器，是感知潮汐的探测器
3. **🌳 根系破土成枝** — 底层思想范式的模块是从根系直接长出的新枝干，跟主干分叉出来的枝干同样有生命力
4. **🍎 果实落回土壤** — 实战案例的复盘成果最终滋养根系，完成认知闭环
5. **理≠觉≠鉴** — 三字天然互斥，不重叠

## 架构评审流程

### Phase 0：理解哲学根基（最重要，必须先做）

在动任何文件之前，花时间理解用户的知识库是**哲学驱动的**，不是功能驱动的。不要假设架构隐喻。

**审查清单（每次架构变更前自检）：**

```
□ 我理解了用户的知识库哲学根基？还是只看了功能描述？
□ 我的隐喻是活的（树）还是死的（楼/层/系统）？
□ 我有没有把「贯穿」误当作「包裹」？
□ 我的变更会不会破坏「两仪互生」的生态？
□ 如果需要，问用户一个问题来验证我的理解，而不是默认正确
```

### Phase 1：现场扫描

1. 枚举所有模块目录 + 文件列表
2. 统计编号冲突、残留旧格式、wikilink 引用
3. 检查各子模块总览文件的一致性
4. 阅读当前 `00-六韬总纲.md` 了解现有架构

### Phase 2：交叉评审

1. 识别问题并打分（每项标严重程度 🔴🟡🟢）
2. 五大评估维度：

| 维度 | 权重 | 评估内容 |
|:-----|:----:|:---------|
| 框架精确度 | 30% | 模块边界是否清晰？是否存在定位矛盾？ |
| 命名一致性 | 20% | 编号是否有冲突？前缀是否统一？发音有无歧义？ |
| 内容可承载性 | 20% | 现有笔记是否能被框架容纳？有无信息丢失？ |
| 使用友好度 | 15% | 用户能否快速找到想要的？ |
| 可维护性 | 15% | 长期增量维护成本高不高？ |

3. 提出修正方案（不要默认嫁接外部理念——先过元原则）
4. 用户确认后进入 Phase 3

### Phase 3：执行变更

1. **先替换内容，再重命名目录** — 顺序不可颠倒。先做文本替换，目录改名放最后。
2. **文本替换三步法**（已验证：道史→史鉴全局更名）：
   - **第一步：全库 sed** — `find . -name "*.md" -not -path "./.obsidian/*" -exec sed -i 's/旧名/新名/g' {} +`。这覆盖路径引用 `[[旧模块/文件]]` 和大部分文字描述。
   - **第二步：跨系统同步** — 对 skill 和 reference 文件执行同样的 sed。路径：`/home/hermes/.hermes/skills/`
   - **第三步：处理残余独立引用** — sed 不会替换「旧名」作为独立词汇的引用（如「主干（旧名）」「旧名是这棵树的主干」「旧名以鉴观世」）。需要针对性追加 sed 模式，或逐个 patch。
3. **目录重命名** — `mv "六韬旧名" "六韬新名"`
4. **更新总纲 + 各子模块总览**（统一树隐喻标题）
5. **更新别名** — 把新名加入总纲文件 frontmatter 的 `aliases` 列表，保留旧名别名作为向后兼容入口
6. **更新 SCHEMA**（分类仲裁树、命名规范）
7. **更新架构提案文档**

**关键陷阱**：
- Obsidian 的 `.obsidian/workspace.json` 会缓存旧路径。**不要改它**——关闭后自动重建。
- 标签（tags）中的旧名引用可以不改——作为历史兼容，保留搜索可达性。
- 旧名 filename（如 `DS-00-道史总纲.md`）可以不改——Obsidian wikilink 解析基于 filename 而非目录名。重命名目录后完整路径 `[[六韬新名/DS-00-旧名总纲]]` 仍能解析，因为文件本体没动。

### Phase 4：验证

1. `grep -rn "旧名" --include="*.md" . | grep -v ".obsidian" | grep -vE "(tags:|aliases:)"` — 确认无残留（排除 tag/alias）
2. `grep -rn "旧名" /home/hermes/.hermes/skills/ --include="*.md"` — 确认 skill 层无残留
3. **旧目录残留检测**（模块更名后新增）：检查旧模块目录是否完全清空，如有孤立文件则迁移到新模块再删除旧目录
4. 确认新前缀引用数正确
5. 确认跨模块链接存活（重要：先文本替换再挪目录，链接才不会断）

### Phase 5：原文覆盖审计（确认已有原文质量）

> **与 Phase 5a 的区别**：Phase 5（分级审计）是对新内容做入仓决策——该不该存原文，存哪一层。  
> **本审计**是对已有内容做验证——声称是 🔴 全文层的文件，是否真的包含了完整原文，还是只有框架/摘要。

#### 触发场景

- 用户怀疑某模块笔记只有框架没有原文（如「六韬易哲板块里的知识都包含了原文吗？」）
- 批量同步 WB→Vault 后检查原文是否真的落地
- 新加入模块后做基线审计

#### 审计四步法

**Step 1：枚举文件清单**

列出目标模块下所有 `.md` 文件，记录文件名和文件大小：

```bash
for f in "path/to/module/"*.md; do
  printf "%-30s %sK\n" "$(basename "$f")" "$(($(wc -c < "$f") / 1024))"
done
```

**大小经验法则（快速判断）**：

| 大小 | 大概率 | 进一步核实 |
|:----:|:-------|:----------|
| >50KB | 🔴 全文层 | 确认开头即为原文（检查前20行） |
| 10~50KB | 🟡 精要层 | 确认是摘录还是完整篇章 |
| 1~10KB | 🟢 框架层 | 确认无原文内容遗漏 |
| <1KB | ⚪ 占位符/空壳 | 标记待补充 |

**Step 2：内容取证（spot-check）**

对标记为 🔴 全文层的文件，随机抽检前30行确认：

```bash
# 确认是否为原文开头（古籍原文→自序/卷一/卦辞等）
head -20 "path/to/module/file.md" 

# 确认全文长度是否合理（全文应有完整目录结构）
wc -l "path/to/module/file.md"
```

**原文特征信号**：自序、校堪序、卷一/上卷、原文开头如「乾：元亨利貞」「第一章 天地玄黄」等。

**框架笔记特征信号**：`---` frontmatter 后紧接 `# 标题` + 表格/列表/```代码块 结构。

**Step 3：三层标记**

| 标记 | 含义 | 行动 |
|:----:|:-----|:-----|
| 🔴 全文层 ✅ | 含完整原文 | 保持 |
| 🟡 精要层 ✅ | 框架笔记，不需原文 | 保持（设计使然） |
| ⚪ 待补充 ⚠️ | 空壳/占位符 | 安排原文入库 |
| 🟡 节选不足 🤔 | 原文不完整，节选不够用 | 评估是否需要全本 |

**Step 4：生成审计报告**

按模块分组，用表格呈现：

```
### 模块名（N个文件）

文件 | 大小 | 层级 | 原文状态
:--- | :--: | :--: | :-------
08-易经经文 | 150K | 🔴 | ✅ 64卦完整+彖传
09-周易正义 | 793K | 🔴 | ✅ 孔颖达注疏完整
01-四柱基础 | 4K | 🟡 | ✅ 框架笔记
千里命稿_待OCR | 1K | ⚪ | ⚠️ 占位符
```

文末给出整体结论：**原文覆盖完好 / 有N处缺口需补充**。

---

### Phase 5a：全库入仓审计（定期维护）

审计步骤：
1. 枚举全库所有模块的文件列表
2. 对每个文件按五维评分判定当前层级（🔴🟡🟢）
3. 标记三类：需补充原文 / 保持框架 / 可删除或合并
4. 执行清理（旧目录/重复文件/冗余文件）

> 参见 multi-bot-persona 流水线⑦：扫描全模块→按五维评分分级→分类标记处理建议。每季度或每次批量入库后执行一次。

### 冲突编号处理

1. 按文件时间戳决定谁保持原号、谁移位
2. 冲突文件取 `max_NN + 1`（永不手动插入中间序号）
3. 全库 wikilink 替换应在重命名**之前**完成

**时事分析子模块说明**（2026-06-06 新增）：当分析对象是**当前/近期的政策、监管、行业动态**（如美国AI行政命令、科技巨头战略调整、国际博弈）时，走 DS- 前缀（因其本质是"以鉴观世"），但在总纲中单独归入「时事分析」树分支，与传统历史/思想笔记区分。归档时文末加「关联思考」小节链接到现有史鉴框架。

交叉归属：主模块按 Q1→Q6 最先命中，副模块标记在 frontmatter `related_modules` 字段。

## 常见陷阱

### 🔴 建筑模型陷阱（P0，最常见）

```
❌ 把知识库画成一栋楼（第一层/第二层/第三层）
✅ 画成一棵树（根系/主干/枝干/交汇/树冠）
```

建筑是静态的、分层的、单向支撑的。树是活的——根系同时向下扎和向上供，还会破土成枝。用户的哲学是两仪互生，不是分层管理。如果画了楼层结构，一定有问题。

### 🔴 双螺旋定位陷阱（P0）

```
❌ 把双螺旋放在包裹层位置（最外层）
❌ 把双螺旋当作「观」步骤的一部分
✅ 双螺旋是根系（最底层），上层建筑都建筑在其上
```

双螺旋是思想范式的地基，不是架构中的「层」。它的模块是「根系破土成枝」——从底层直接长出的新枝干。

### 🟡 物我合一分身陷阱

```
❌ 认为 ZN-19 和 YZ-物我合一是同一篇笔记分身
✅ ZN-19 是决策方法论版（18KB），YZ-物我合一是哲学思考版
```

两个版本的笔记内容不同、用途不同，各归各位。

### 🟡 百年孤独/一句顶一万句分身陷阱

```markdown
❌ 认为 DS-16/18 和 易哲/文史哲下的同名笔记是分身
✅ DS-16/18 是深层分析（11-12KB），易哲版是阅读笔记
```

深度不同，各归各位。总纲标注：正本在史鉴·阅读笔记在易哲。

### 🟡 模块命名发音歧义陷阱

```markdown
❌ 选字只考虑字面意思：道史 = 历史之道
✅ 选字同时考虑发音歧义：史鉴 = 以史为鉴，不串音
```

**教训案例**：原模块名「六韬道史」字面意思是「历史之道」，但 dào shǐ 读快了容易串成 dào shì（道士）甚至 dào shǐ（倒屎）。用户指出后改为「六韬史鉴」，发音干净无歧义。

**命名检查清单**（新增两条）：

| # | 检查项 | 示例 |
|:-:|:------|:-----|
| 1 | **字面意思是否准确？** | 道史=历史之道 ✅ |
| 2 | **发音是否有歧义？** | 道史≈道士/倒屎 ❌ |
| 3 | **与其他模块名是否同一风格？** | 智脑/易哲/史鉴 三字对仗 ✅ |
| 4 | **读出来是否利落？** | 六韬史鉴 四字顿挫 ✅ |

**原则**：模块命名是知识库的公共接口——用户在对话中、笔记中、思考中都会念出它。名字是工具，不是诗。宁选字面普通但发音干净的名字，不选字面精准但读起来有歧义的名字。

### 🔴 WSL路径中文目录操作陷阱

在WSL环境下对 `/mnt/c/Users/...` 路径中的**含中文的目录**执行 `rm -rf` 时，命令可能超时失败（`BLOCKED: Command timed out`）。原因：WSL通过DrvFs访问Windows文件系统，含中文/特殊字符的路径在跨文件系统操作时可能触发锁或性能问题。

**修复**：用 Python `shutil.rmtree()` 替代 shell `rm -rf`：
```python
import os, shutil
path = "/mnt/c/Users/.../六韬道史"
if os.path.exists(path): shutil.rmtree(path)
```

**也影响**：`cp`/`mv` 含中文文件名的文件。cp成功但rm超时 = 文件出现在两地。推荐 `shutil.copy2()` + `os.remove()` 组合确保安全迁移。

## 新内容归档工作流（飞书文档→知识库）

重复出现的模式：创建/完成飞书文档后，归档到 Obsidian 知识库并更新模块总纲。三步走：

### Step 1：创建归档笔记

在对应模块目录下创建 `DS-NN-主题.md`（或 `ZN-`/`YZ-` 等模块前缀），笔记包含：
- **Frontmatter**：title、tags（模块+主题）、aliases、created（当日日期）
- **正文首行**：飞书云文档链接（用 `>` 引用块包裹）
- **关联笔记**：用 `[[wikilink]]` 链接到已有相关笔记
- **返回链接**：文末 `*返回 [[DS-00-道史总纲]]*`

### Step 2：归档笔记模板

```markdown
---
title: 主题·读书笔记
tags:
  - 六韬史鉴
  - 主题标签
aliases:
  - 别名1
created: YYYY-MM-DD
---

# 主题·读书笔记

> 飞书云文档：[标题](https://my.feishu.cn/docx/TOKEN)

---

## 内容框架（用表格简要总结）

## 🔗 关联笔记

- [[DS-XX-已有相关笔记]]

*返回 [[DS-00-道史总纲]]*
```

### Step 3：更新模块总纲

1. 在 `DS-00-道史总纲.md` 的模块地图中添加新条目（`│   ├── [[DS-NN-主题]]    —— 一句话说明`）
2. 在关联表格中添加新行
3. 更新 frontmatter 的 `updated:` 字段为当日日期

### 命名规则
- 编号：取 `max_NN + 1`（永不手动插入中间序号）
- 前缀：对应模块前缀（DS-史鉴 / ZN-智脑 / YZ-易哲 等）
- 文件名：`DS-NN-主题.md`，主题用中文

### Step 4：更新搭建日志 Bitable

归档完成后，在搭建日志表（Bitable: `tbllV9WgN64Zwput`）新增一条记录。用 bot 身份写入：

```bash
lark-cli base +record-upsert --base-token NzuPbMtMFa0wUusVQKwc69lenib --table-id tbllV9WgN64Zwput --as bot --json '{"时间":"YYYY-MM-DD","操作内容":"...","领域":"知识库基建","状态":"✅ 已完成","沟通状态":"○ 无需沟通","备注":"归档至DS-NN"}'
```

> **这是一条强制规则。** 所有飞书文档创建/知识库归档操作结束后必须执行此步骤，用户会检查搭建日志是否有遗漏。

## 🔗 跨模块交叉链路巡检（Cross-Module Link Audit）

> **场景**：知识库不断扩建后，模块间的交叉引用链路退化——旧笔记引不到新模块，新内容无人问津。这是架构维护的**常态化巡检**，不是一次性重构。

### 触发词

> 「链路分析」「交叉引用巡检」「跨模块关系」「断链检测」「链接审计」「黄老邪」

### 五步巡检法

#### Step 1：枚举目标模块

列出要巡检的模块下的所有 `.md` 文件：

用 `search_files(pattern="*.md", target="files", path="...")` 直接遍历目标模块路径。

#### Step 2：扫描既有链路（双向）

**正向扫描**（模块内 → 模块外）：搜索每个笔记中的跨模块 `[[其他模块/...]]` 引用。

**反向扫描**（其他模块 → 本模块）：用 `search_files(target="content", pattern="模块名或关键文件名")` 在整个 vault 中查找引用。

#### Step 3：链路健康检查

每条链路标记一个状态：

| 状态 | 含义 | 行动 |
|:----:|:-----|:-----|
| ✅ **健全** | 链接正确、目标存在 | 保持 |
| ⚠️ **可优化** | 存在但内容单薄或仅有索引层引用 | 考虑追加具体锚段链接 |
| ❌ **断链** | 链接目标不存在、路径前缀错误、文件名不对 | 修复路径 |
| 🔵 **缺失** | 应有链路但不存在 | 新增链接 |

**常见断链模式**（实战教训）：

| 模式 | 示例 | 修复 |
|:----|:-----|:-----|
| 路径缺前缀编号 | `[[../中医基础/藏象学说]]` → 应 `[[../中医基础/03-藏象学说]]` | 补齐 `NN-` 前缀 |
| 别名替代真实路径 | `[[../中医基础/易医同源哲学]]` → 应 `[[../中医基础/01-易医同源哲学]]` | 用实际文件名 |
| 跨级路径计算错误 | 引用中 `../` 层级不对 | 从目标文件相对路径推算 |
| 文件已重命名但旧链接残留 | 模块更名后旧 wikilink 未批量替换 | 全库 sed |

#### Step 4：缺失链路推断（最大价值）

基于内容语义推断应该存在但缺失的链路。判断依据：

| 证据类型 | 说明 | 示例 |
|:---------|:-----|:-----|
| **共用体系** | 两个模块共享同一套概念/符号系统 | 五运六气与八字命理共用干支体系 |
| **同源分支** | 两个概念出自同一哲学根基但分布于不同模块 | 中医五色与面相五色同源（五行色） |
| **医理互参** | 命理/相理的诊断参数可作为中医辨证的补充依据 | 日主旺衰与体质倾向 |
| **工具共享** | 同一张表/同一套数据同时在多个模块中出现 | 十二长生表同时出现在五运六气与八字命理 |

**每发现一对缺失链路，给出：**
- 源文件 → 目标文件
- 链接理由（一句话）
- 建议插入位置

#### Step 5：输出巡检报告

报告结构模板：

```markdown
## 跨模块交叉链路巡检报告

### 一、现状诊断
（汇总统计数据）

### 二、需修复的断链
（位置 → 问题 → 修复方案）

### 三、推荐新增链路（按模块排列）
（源文件 → 目标文件 → 链接理由 → 插入位置）

### 四、MOC/总纲更新建议
（列出需同步更新的 MOC 地图或总纲文件）
```

### 关键陷阱

- ⚠️ **不要只做正向扫描**——反向扫描（其他模块引用本模块）往往发现更多断链
- ⚠️ **五运六气类内容往往是最大桥梁孤岛**——与八字命理共用干支、与易经共享河洛数理，却最容易缺乏跨模链接
- ⚠️ **不要遗漏总纲/MOC文件**——它们是新用户的入口，链路必须健全
- ⚠️ **别名别名陷阱**——用户在 frontmatter `aliases` 中设定的别名不会自动创建 wikilink，必须手动确认目标文件是否存在
- ⚠️ **区分『别名引用』与『文件路径引用』**——用户有时用别名而非文件路径创建链接（如 `[[易医同源哲学]]` 代替 `[[01-易医同源哲学]]`），这在 Obsidian 中能解析，但在知识库迁移/统一前缀时会断裂。

### 执行流水线：跨模块链路巡检的协同作业

> 2026-06-04 首次全流程验证通过。四道工序咬合，产出质量 B+。

链路巡检不是一人能完成的——需要多Bot协作流水线：

| 工序 | Bot | 职责 | 产出物 |
|:----:|:---|:-----|:-------|
| 1 🟢 分析 | **黄老邪** | 读源模块所有笔记 + 目标模块扫描 → 输出交叉链路表 | 链路分析报告（含现状诊断+修复+新增建议） |
| 2 🔧 执行 | **燕青** | 根据分析报告执行文件修改（修复断链+新增链接段落） | 12+个文件的 patch/追加 |
| 3 🎨 润色 | **鲁班** | 更新MOC地图、检查链接描述流畅度、更新总纲 | MOC更新+质量抽查 |
| 4 🧘 校验 | **扫地僧** | 哲学一致性校验（两仪互生原则） | 校验报告（剔除关联薄弱的链接） |

**关键经验（2026-06-04实战，两轮累积）：**

**燕青执行阶段**：用 `delegate_task` 让燕青子代理处理所有文件修改。35条链接 + 5处修复一次完成。使用 `terminal` 的 shell 命令验证结果（grep确认无残留）。

**第二轮巡检发现的问题（智脑→全模块，156条建议→实际执行77条）：**
- **模块密度差异**：纯方法论模块（智脑）的跨模块链接密度天然低于医易同源模块（中医基础）。智脑→史鉴最密集，→易哲最稀疏。
- **报告验证环节缺失**：黄老邪分析报告中部分链接目标不存在（如易哲单卦文件名不符、史鉴文件路径错误），燕青执行时需逐一验证，耗费大量时间。
- **改进方案**：燕青应在黄老邪输出报告后先跑一轮`grep`验证所有链接目标是否存在，将不存在目标标记为红色返回给黄老邪修正，再执行。这样减少燕青执行时的验证负担。

**改进后的流水线**：
| 工序 | Bot | 职责 |
|:----:|:---|:------|
| 1 | 黄老邪 | 分析+输出建议链接表 |
| → | **新增: 燕青** | **前置验证：跑grep确认所有链接目标存在** |
| 2 | 燕青 | 执行（已验证的链接） |
| 3 | 鲁班 | 润色+更新MOC |
| 4 | 扫地僧 | 哲学校验 |

**扫地僧校验发现的问题模式**（可复用于后续巡检）：

| 问题编号 | 链接 | 扫地僧发现的问题 | 处理 |
|:--------:|:-----|:----------------|:----:|
| P1 | 气血津液→五宫星象 | 紫微斗数星煞 ≠ 五运六气五星气化，两套体系哲学根基不同 | 改为 `天机道/02-五星运行` |
| P2 | 九种体质→大运流年 | 静态体质分类 ≠ 动态运势推演，缺乏中医学理支撑 | 删除 |
| P3 | 九种体质→地理风水 | 风水方位 ≠ 内经异法方宜论，泛化天人感应 | 删除 |
| P4 | 养生要诀→地理风水 | 主动行为养生 ≠ 居住环境风水吉凶，跨越过大 | 删除 |
| P5 | 四诊详解→四柱基础 | 天干面部对应是面相学概念，非四诊内容 | 删除 |

**核心原则**（从扫地僧校验经验中总结）：
1. **共用体系优先**——直接共享字符/概念体系的模块优先链接（如五运六气与八字命理共用干支）
2. **医理互参优于泛化链接**——有经典文献依据的链接（如《黄帝内经》异法方宜论）优于泛化的天人感应
3. **静态概念不链动态工具**——相对稳定的概念（体质类型）不链接动态推演工具（大运流年），除非有明确的转化机制

## 原文内容审计

当用户问"XX板块有没有原文"时，执行三步审计：

### Step 1: 扫描目录
列出模块下所有文件及大小：
```bash
for f in "path/to/module/*.md"; do printf "%-25s %sK\n" "$(basename "$f")" "$(($(wc -c < "$f") / 1024))"; done
```

### Step 2: 分级判定
| 大小 | 大概率 | 举例 |
|:----:|:------|:-----|
| >50KB | 🔴 全文层 | 经文、注疏、整部典籍 |
| 5-50KB | 🟡 精要层 | 读书笔记、摘录 |
| <5KB | 🟢/⚪ 框架层 | 提纲、目录 |

### Step 3: 抽样验证
对标志性的中等文件（50-100K）和关键框架文件，head -20 确认内容类型。
完整审计报告格式：表格列明每个文件的大小/层级/原文状态。

**⚠️ 避坑**
- 大文件未必全文（可能只是单章译文），小文件未必无原文（可能是原文片段）
- 易哲模块的倪海厦天纪框架空壳（1-3K）是已知的等待文渊匠状态

## 原文与框架分层决策：五维评分体系

> 什么知识存原文，什么知识只提炼框架？五维评分，三级入仓。

### 五维评分

每次入库前按五个维度打分：

| 维度 | 权重 | 高分特征（3分） | 低分特征（0分） |
|:----|:----:|:---------------|:---------------|
| **源典性** | 30% | 源典/原创经典，后世注疏以此为本 | 现代教材/个人解读/二次创作 |
| **密度** | 25% | 每句话不可替代（卦辞/歌诀/经方） | 可概括为几句话的论述性内容 |
| **参考频次** | 20% | 高频查询工具（速查表/方剂/卦象） | 读一次即懂的背景知识 |
| **跨模块度** | 15% | 两个以上模块共用（干支/五行/经脉） | 单个模块自洽的知识 |
| **体量** | 10% | 短小精悍（<10万字），值得全本入库 | 鸿篇巨制，存框架即可 |

**总分 = Σ(维度分 × 权重)**

### 三级入仓

| 总分区间 | 层级 | 操作 |
|:--------:|:----:|:------|
| **≥2.5** | 🔴 **全文层** | 存完整原文，编入典籍目录 |
| **1.5~2.4** | 🟡 **精要层** | 存核心片段+框架+白话精要 |
| **<1.5** | 🟢 **框架层** | 存结构大纲+核心主张+关联笔记 |

### 核心原则

> **宁可精要不全文，不可全文不精要。**
> — 全文层的笔记必须附一段精要摘要（为什么重要、核心是什么）。
> — 框架层的笔记必须附一条关联链路（在哪个模块、跟什么相关）。

### 🎯 用户偏好优先级（五维评分平局时的决胜规则）

**知识领域优先级**（用户明确表达，按降序排列）：

| 优先级 | 领域 | 示例 |
|:------:|:-----|:------|
| **P0** | **历史** | 晚清人物/改革开放/毛泽东邓小平/通鉴资治 |
| **P0** | **经济学** | 黄奇帆/曾鸣/宏观微观/制度经济学 |
| **P1** | **易经**（近期深耕） | 64卦经/周易正义/梅花易数/六爻 |
| **P2** | **中医/命理/相学** | 内经/滴天髓/麻衣神相（合入分数取齐） |

**何时应用**：当两个候选知识点五维评分相同（差距 < 0.2 分）时，按上述优先级选取更高优先级的领域存原文/精要。

**实战案例**：冰鉴-曾国藩（37KB, 历史人物）vs 麻衣神相（23KB, 相学），评分同为2.2分→按优先级选 冰鉴（历史P0 > 相学P2）先行入库。

### 速查案例

| 经典 | 总分 | 层级 | 处理 |
|:-----|:----:|:----:|:------|
| 易经经文 | 3.0 | 🔴 全文 | 64卦经+十翼 |
| 道德经 | 2.8 | 🔴 全文 | 5千字体量极小 |
| 滴天髓 | 2.5 | 🔴 全文 | 34章诗诀+任氏注 |
| 黄帝内经 | 2.25→🔴(特殊) | 源典3分特殊处理 | 分篇结构可承受 |
| 王阳明传习录 | 2.2 | 🟡 精要 | 存核心语录+深研笔记 |
| 中医基础教程 | 1.0 | 🟢 框架 | 结构大纲+速查表 |
| 八字命理教程01~06 | 1.0 | 🟢 框架 | 方法论框架 |
