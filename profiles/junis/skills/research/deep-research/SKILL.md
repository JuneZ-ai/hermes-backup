# 深度研究（Deep Research）Skill

> 吸收自 AutoGLM DeepResearch v1.0.0，适配 Hermes 原生工具链。
>
> **核心理念：够用即可。** 信息过量比信息不足更常见，限制调用数量倒逼思考质量，而非牺牲质量。

---

## 附加原则：吸收评估（Absorption Assessment）

当评估外部资源（技能/文章/工具）是否有价值时，走以下流程：

```
① 这是什么？(类别判断：方法论/工具/产品/噪音)
② 核心创新点？（跟已有体系有什么不一样）
③ 能吸纳什么？（模式>工具，方法论>安装）
   → 有独特模式 → 吸收到对应 skill
   → 跟已有重叠 → 不装
   → 只改流程/行为 → 不建新 skill
   → 需要新 skill → 创建。
```

**核心立场：吸收但不安装。** 方法论和设计模式比工具和 API 更有长期价值。工具会被更新换代，模式不会。今天装的工具半年后可能不用了，但今天吸收的模式半年后还在用。

## 附加原则：文件即状态（File-as-State）

> 吸收自 pi-coding-agent（Mario Zechner）

研究的中间结果、搜到的关键页面、思考链条——**不要只留在对话流里，写到文件。**

| 时机 | 写什么 | 写到哪 |
|:----|:------|:-------|
| Step 2 搜索完 | 搜索结果摘要 | `~/.hermes/resources/research/<课题>/search_results.md` |
| Step 3 读完页面 | 页面关键信息 | `~/.hermes/resources/research/<课题>/page_notes.md` |
| Step 4 中间发现 | 已确认/待核实/缺口 | `~/.hermes/resources/research/<课题>/interim_findings.md` |
| Step 5 最终报告 | 完整报告 | 如果用户后续要入库 → Obsidian 笔记 |

**为什么这么做：**
- 跨 session 可引用 — 用户明天问"上次那个研究"，文件还在
- 可版本化 — 研究过程本身是可复现的
- 可重用 — 中间产物可以被另一个课题引用
- 文件比对话更便宜 — 对话历史有 retention，文件不会丢

**约束：** 一次研究的中间文件不超过 5 个。不要再开新文件了。

## 与韬定律的关系

| 标准 | 对应原则 |
|:----|:---------|
| 做对的事 > 做更多的事 | 少搜少读，聚焦关键问题 |
| 改变连接方式 > 加新节点 | 把已有信息交叉分析，比搜更多页面有价值 |
| 修三次不好 → 换维 | 同一方向搜两次还不出结果，换搜索词/换角度 |

## 执行流程

### Step 1：拆解子问题 🔍

**硬约束：分成 1～2 个搜索方向，最多 2 个。**

不要为了「完整性」拆出 5 个方向。优先选最能直接回答问题的维度：

- **背景与定义** — 这是什么？为什么重要？
- **最新现状/关键数据** — 现在什么情况？数字是多少？
- **代表性案例** — **仅在确有必要时才加**，不是标配

如果用户的问题本身已经很具体（比如「某公司 2026 年营收」），直接进入搜索，不必再拆。

### Step 2：多轮搜索 🌐

**硬约束：最多调用 web_search 2 次。**

- **默认只搜 1 次**。选最精准的关键词。
- **第 2 次搜索的条件**（三者至少满足其一）：
  - 第 1 次结果明显不够
  - 结果质量差（相关性低、来源不可靠）
  - 缺少关键维度
- 每次搜索后，先展示中间结果再决定是否继续

**中间结果格式：**
```
📡 搜索词：<关键词>
📎 最相关结果：
  1. [标题](url) — snippet
  2. [标题](url) — snippet
  ...
💬 判断：信息足够 / 需要补充 / 方向需调整
```

### Step 3：深度阅读 📖

**硬约束：最多读 3 页（默认 1 页）。**

筛选标准：
- snippet 信息量丰富、与课题高度相关
- 来源权威（官网、知名媒体、学术机构、一手数据源）
- 避免重复来源（不同媒体报道同一件事 = 1 条）

**规则：**
- 默认只打开 1 个页面
- 仅当第 1 个页面信息不足、存在缺口、或需交叉验证时，才加第 2 或第 3 页
- 不批量打开多个相似来源

**每次深读后展示中间提炼：**
```
📄 页面：<标题>
🔑 关键信息点：
  1. ...
  2. ...
  3. ...
💬 相关性：直接相关 / 部分相关 / 作为背景
💬 还有缺口吗：有 → 继续读 / 够了 → 进入总结
```

### Step 4：中间发现先行

在出最终报告前，先列一段：

```
## 中间发现
- ✅ 已确认：...
- ❓ 待核实：...
- ⚠️ 信息缺口：...
- 📊 主要来源：...
```

这步的价值：**让用户知道哪些结论是稳的、哪些还不确定，而不是包装成一切确定。**

### Step 5：输出结构化报告

```
# [课题名称] 深度调研报告

## 概述
2～3 句话概括核心结论

## 背景
基本定义、背景信息、为什么值得关注

## 现状分析
关键数据、现状描述、主要玩家/趋势

## 典型案例 / 代表性观点
具体案例或多方观点（如是多面争议的话题）

## 发展趋势
未来走向判断、机会与风险

## 总结
综合结论与可行建议（如适用）

## 参考来源
1. [标题](url)
2. [标题](url)
...
```

## 执行自查

提交报告前自我检查：

- [ ] `web_search` 调用了 ≤2 次？
- [ ] `web_fetch`/页面阅读 ≤3 次？
- [ ] 每一轮搜索/阅读都展示了中间结果？
- [ ] 报告前有一段「中间发现」？
- [ ] 报告结构完整（概述→背景→现状→案例→趋势→总结→来源）？
- [ ] 是否在信息足够时就提前停止，没有为「完整性」多搜？
- [ ] 结论清楚区分了「确认的」和「推测的」？

## Pitfalls

⚠️ **不要为了「看起来完整」而多搜。** 如果 2 次搜索 + 1 页深度阅读已经够回答用户问题，立即进入总结。
⚠️ **不是每次都要用全所有步骤。** 如果用户的问题只需搜一次就能答，Step 2 做完直接跳到 Step 5。
⚠️ **不要合并多步中间结果。** 每搜完一轮、每读完一页，都要展示中间结果——这是跟用户的对话节奏，不是预处理的内部步骤。
⚠️ **「中间发现」不是报告的草稿。** 它是独立输出，告诉用户「到目前我看到了什么、还缺什么」，然后用完整报告回答缺口。
