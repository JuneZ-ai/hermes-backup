# 多Agent团队设计模式

> 吸收自 `multi-agent-team-design`（已归档）。补充 multi-bot-persona 流水线架构未覆盖的团队设计原则和模式。

## 核心哲学

> Bot之间的分工，不是「各干各的」，而是「一条流水线上的不同工位」。

每个 Bot 的能力圆必须**有明确的缺口**——缺的正是其他 Bot 补的。没有缺口的分工是重复造轮。

## 工具集精确原则

> 不要多，要精准。多一个工具比少一个更糟糕。

- 哲学思辨Bot = 不要 terminal（它不需要跑脚本）
- 执行Bot = 不要 web search（它不需要查资料）
- 分析Bot = 不要 cron（它不需要定时任务）

决策方法：
1. 先想清楚这个角色「做且只做」哪一类事
2. 只给完成那类事**刚好够用**的工具集
3. 凡是拿不准的、可能用不到的，一概禁掉
4. 以后发现不够再加，但绝不提前配冗余

## 模块密度差异

不同模块的跨模块链接天然密度不同，影响巡检时间和人力投入：

| 模块类型 | 特征 | 建议人均链接数 | 示例 |
|:---------|:-----|:--------------:|:-----|
| 医易同源型 | 跨学科天然交叉 | 3-5条/笔记 | 中医基础、决断之桥 |
| 方法论型 | 同模块自洽为主 | 2-3条/笔记 | 六韬智脑 |
| 历史案例型 | 与史鉴/智脑双向密集 | 2-4条/笔记 | 六韬史鉴 |
| 底层范式型 | 与所有模块有间接连接 | 2-4条/笔记 | 太极双螺旋 |

## 流水线执行经验（修正后）

### 修正冲突：黄老邪的链接目标不可靠

**现象**：智脑批次中，黄老邪建议的156条链接约有50%的目标文件不存在。

**根因**：黄老邪在分析时根据**内容语义**推断目标文件名，而非根据**文件系统真实路径**。

**修正后流水线**：

```
黄老邪(分析) → 燕青(前置验证+标记不存在目标) → 黄老邪(修正) → 燕青(执行) → 鲁班(润色) → 扫地僧(校验)
```

| 工序 | Bot | 职责 |
|:----:|:---|:------|
| 1 | 黄老邪 | 输出建议链接表 |
| 2 | **燕青** | **跑 grep 验证所有目标文件是否存在，返回红色清单给黄老邪** |
| 2b | 黄老邪 | 根据红色清单修正目标路径或删除不存在条目 |
| 3 | 燕青 | 执行已验证的链接 |
| 4 | 鲁班 | 润色+更新MOC |
| 5 | 扫地僧 | 哲学校验（每30条剔除约5条弱关联） |

## 角色边界定义方法

### 每Bot的黄金三问

1. **做且只做什么？** — 能力圆的圆心
2. **绝对不做什么？** — 能力圆的边界（比「做什么」更重要）
3. **交棒给谁？** — 超出边界时的默认路由

## @mention Group Chat 前置条件

多飞书应用方案必须满足：
- 每个Bot在飞书开放平台注册为独立应用
- 每个Bot的Gateway进程独立运行
- **每个Bot的.env必须包含 `FEISHU_BOT_OPEN_ID` 和 `FEISHU_BOT_NAME`**（@mention检测依赖这两个字段，漏配则所有@被无声丢弃）
- **Bot必须由人类用户在飞书客户端内手动添加进群**（API无法将一个Bot通过app_id加另一个Bot的群）
- 入群验证通过 `chat.get` 的 `bot_count` 字段（非 `chat.members.member_total`）
