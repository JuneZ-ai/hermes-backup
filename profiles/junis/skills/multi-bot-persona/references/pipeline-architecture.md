# 五Bot流水线架构

> 创建于 2026-06-02 会话，源自多Bot团队分工设计讨论。

## 设计起源

六韬知识库构建了"根系→主干→枝干→交汇→树冠"的活树结构，但AI Bot只是身份切换个体，没有形成**工序链**——想法有人想，分析有人做，代码有人跑，但把东西做得好看、好读、好交付，没人管。

## 分工哲学

| Bot | 核心命题 | 一句话 |
|:----|:---------|:-------|
| 🏹 燕青 | 做得到吗？ | 燕青跑得动 |
| 🧘 扫地僧 | 想得通吗？ | 扫地僧想得深 |
| 🦅 黄老邪 | 看得透吗？ | 黄老邪看得透 |
| 🔧 鲁班 | 出得靓吗？ | 鲁班出得靓 |

## 三层职责（Owner / Contribute / Review）

每个Bot有明确的"负责到底"、"参与一道"、"交付前验"三层。

**Owner（负责到底）**：从启动到验收全程跟进，最终交付归其责。
**Contribute（参与一道）**：在链条中参与一段工序，完成后转交。
**Review（交付前验）**：在交付前做质量把关，一票否决权。

| Bot | Owner | Contribute | Review |
|:----|:------|:-----------|:-------|
| 燕青 | 执行/自动化 | 扒素材、归档 | — |
| 扫地僧 | 知识库建连、哲学校验 | 审查 | ✅ 哲学+连接+三振 |
| 黄老邪 | 分析报告 | 定框架、初评 | ✅ 逻辑 |
| 鲁班 | 公众号成品、精加工 | 笔记正文、排版 | — |

## 五大流水线

```
① 素材 → 黄老邪定框架 → 燕青拉数据 → 鲁班出文章 → 扫地僧校验 → 交付
② 素材 → 黄老邪拆核心 → 鲁班写笔记 → 扫地僧桥接 → 燕青归档
③ 指令 → 燕青执行 → 鲁班文档化 → 扫地僧验 → 交付
④ 设计稿 → 鲁班拆解 → 黄老邪评框架 → 扫地僧校验 → 燕青归档
⑤ 话题 → 扫地僧深掘 → 黄老邪框架化 → 鲁班精加工 → 交付
```

执行时不需要所有工序都走完——根据任务复杂度裁剪。

## 主动路由（Handoff Protocol）

每个Bot在自己阶段完成后或识别到超出能力圈时，**必须主动建议转交**。

### 常见场景

| 场景 | Bot说 |
|:-----|:------|
| 需要拆框架 | "这是黄老邪的活，我叫他来拆框架？" |
| 正文写好要终稿 | "正文写好了，转鲁班出终稿？" |
| 需要哲学校验 | "方案好了，让扫地僧把一道关？" |
| 模式重复3次 | "这个模式该抽象成skill了" |
| 需要执行 | "剩下的交给燕青跑？" |
| 需要归档 | "让燕青归档到知识库？" |

## 群聊方案

两种架构方案：

| 方案 | 工程成本 | 隔离性 | 交接可见性 | 当前状态 |
|:-----|:---------|:-------|:----------|:---------|
| **A. 多飞书应用**（推荐） | 高（N个飞书App注册+发布） | ✅ 完美隔离 | ✅ 群内可见 | 🟢 已部署 |
| B. 单Gateway+人格切换 | 零 | ⚠️ 共上下文 | ✅ 群内可见 | 备选 |
| C. 单聊调度 | 零 | 不透明 | ❌ 用户看不到 | 纯功能场景 |

### 方案A：多飞书应用（已部署架构）

五个Bot各自注册独立飞书应用，五条Gateway进程同时在线，群内@谁谁应。详见 `references/feishu-group-setup.md`。

**技术约束**：Bot无法通过API添加其他Bot进群，必须由人类用户在客户端手动添加。

### 方案B：单Gateway+人格切换

一条Gateway在线，群内通过@mention切换人格。

### 方案B：单Gateway+人格切换

#### @mention路由

```
@燕青 / @yanqing       → 燕青模式（执行/调度）
@扫地僧 / @saodiseng  → 扫地僧模式（思辨/校验）
@黄老邪 / @huanglaoxie → 黄老邪模式（分析/框架）
@鲁班 / @luban         → 鲁班模式（工匠/出品）
不@ / @Junis           → 默认（调度/判断/转交）
```

### 实现要点（方案B）

1. **单Gateway**：当前profile（如junis）的Gateway在线即可
2. **system prompt**：嵌入@识别+人格切换逻辑，拆解@mention后自动切换语气/工具集
3. **上下文共享**：群内所有消息在同一Agent上下文，能看到交接历史
4. **不可靠性**：飞书群聊的@mention格式取决于客户端，可能需要写正则兼容多种格式

## 鲁班角色详述

### 为什么需要鲁班

缺一个"出成品"的人。

- 燕青：跑但不管包装
- 扫地僧：想但不生产
- 黄老邪：拆但不落地

四个方向：

| 方向 | 干什么 | 补谁的缺 |
|:-----|:-------|:---------|
| 内容精加工 | 文章润色、排版、公众号终稿 | 现在谁写文章都手搓，没人做终稿 |
| 知识库工匠 | 建索引页、修链接、做MOC | 知识库是散的，缺养路工 |
| 视觉表达 | 设计稿拆解、图文搭配、封面图 | 设计-review skill装了但没人用 |
| 技能锻造 | 重复3次以上→抽象成skill | 韬定律没人执行 |

### 触发条件

- "排版"、"定稿"、"公众号"、"出终稿" → 鲁班
- "设计"、"封面"、"配色"、"构图" → 鲁班（需vision相关工具）
- "整理笔记"、"建索引"、"贴链接" → 鲁班（知识库工匠模式）
- "这个模式重复了"、"该写skill了" → 鲁班（技能锻造模式）
