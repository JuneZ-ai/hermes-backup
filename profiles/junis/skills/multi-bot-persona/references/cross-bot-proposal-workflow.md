## 关键模式

### 0. 抽象层级：提案的"显性层 vs 隐性层"

FDE提案实战的扫地僧审查发现了一个系统性问题：**主干三层（L1/L2/L4）都有，但承重梁（L3工具层、L5反馈层）被当作"不着急的东西"跳过了**。这个坑在并行分工阶段不可能暴露——因为每路都在自己的边界内完美执行。

**原则：任何一个框架/系统/架构类交付，都要做「显性/隐性双层扫描」。**

- **显性层**（客户看得懂、说得出的部分）：输入→处理→输出
- **隐性层**（决定系统能不能长大的部分）：标准/基建（L3）+ 反馈闭环（L5）+ 安全约束

扫描方法：把框架画出来 → 标出"看的见的层"和"看不见的层" → 问"没有这一层，系统还能活吗？"

扫地僧在FDE提案中补的2.4节（树→架构映射）和3.3节（信息安全分级），本质上是把隐性层显性化了。

### 1. 并行delegate_task分工

使用 `delegate_task` 的 `tasks` 数组模式同时发送3个子任务：

```python
delegate_task(tasks=[
  {"goal": "生成feishu_bot.py", "toolsets": ["terminal","file"]},
  {"goal": "竞品对标+定价分析", "toolsets": ["web","search","file"]},
  {"goal": "补充两个附录", "toolsets": ["file","web"]}
])
```

### 2. 合稿要点

- 先确认每路产出质量（读取关键文件验证）
- 扫地僧的审查意见不是修正拼写/措辞，是**结构性补丁**——缺什么章节、什么概念没对齐
- 品牌命名（"包打听·问ta，都知道"）适合在结构定稿后作为最后一步

### 3. 提案文档结构模板

```
1. 客户痛点与解决方案
2. 核心理念（视觉隐喻）
3. 架构模型（与隐喻对齐）
4. 核心功能/范式
5. 交付体系/AI配置
6. 实施路径 + 案例
7. 客户适配指南
8. 交付流程
9. 报价（三级锚定）
10. 竞品对比
11. 诚实披露
12. 交付包结构
+ 附录（技术文档/code/视觉图）
```

### 4. 定价心理学模式

- 森林包¥98K（锚点）→ 成树包¥48K（靶心/折中效应）→ 育苗包¥12.8K（诱饵）
- 不按人头（区分SaaS竞品），按方法论价值定价
- 价格速查表（15人或20人规模对比所有竞品）

### 5. 结构审查报告格式（改动复盘表）

扫地僧交付审查意见后，Junis应整理为结构化对比表回给用户确认。格式：

```
| 改动 | 性质 |
|------|------|
| 新增2.4节：三层→五层架构映射 | ✅ 补缺（桥接隐喻与架构的逻辑断层）|
| 新增3.3节：信息安全分级 | ✅ 补强（堵住了企业交付的安全缺口）|
| domain字段从5个精简到3个 | ✅ 精简（与L2三层对齐，消除冗余）|

四类标签：补缺 / 补强 / 精简 / 精调
```

这样用户扫一眼就知道每处修改是什么性质，不需要逐句比对新旧版全文。

扫地僧审查后，综合改动如下。格式可复用：

### 7. 交付物自检清单（审查后）

- [ ] 视觉隐喻与架构模型之间有映射桥梁
- [ ] 信息安全/权限管理有明确章节
- [ ] 条件/边缘情况有说明
- [ ] 跨节引用一致
- [ ] 定价基准与目标客户匹配

## Pitfalls

- ⚠️ 并行子任务只看自己那一块——合稿后必须有结构性审查环节
- ⚠️ 竞品分析中提到的价格要注明估算依据，不要硬写死数字
- ⚠️ 附录文件路径在文档中被引用后，实际文件不存在会失信——确保路径对应真实文件
- ⚠️ 品牌命名在提案过程中的时机：结构未定时过于提前命名会导致返工，结构定稿后命名是一次性的点睛之笔
