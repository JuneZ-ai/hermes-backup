## 搜索工作流（Jannis调度核心能力）

搜索是Jannis调度的底层能力，所有Bot在执行前可能需要搜索外部信息。按优先级使用以下渠道：

### 搜索渠道优先级

| 层级 | 渠道 | 配置状态 | 用途 |
|:----:|:-----|:--------:|------|
| 🥇 **L1** | **Tavily API**（`web.search_backend: tavily`） | ✅ 已配置 | 通用网页搜索、信息查找、竞品调研。1-3秒返回结构化结果 |
| 🥈 **L2** | **Tavily MCP**（search/extract/map/crawl工具） | ⏳ 待重启 | 需要提取特定URL内容、分析网站结构、爬取多页面时使用 |
| 🥉 **L3** | **curl兜底链**（`references/curl-search-workaround.md`） | ✅ 可用 | 中文内容、微信公众号、知乎等非公开API的搜索场景 |
| 🅿️ **L4** | **delegate_task子代理搜索**（带"web"/"search"工具集） | ✅ 可用 | 需要大规模或深度搜索时，交给子代理并行执行 |

### Tavily 搜索使用规范

**配置确认：**
```yaml
# config.yaml
web:
  search_backend: tavily
mcp_servers:
  tavily:
    command: npx
    args: ['-y', 'tavily-mcp@latest']
    env:
      TAVILY_API_KEY: <已在.env中配置>
```

- 免费额度: 1000次/月
- 响应时间: ~1-3s
- 搜索失败降级到L3 curl兜底链
- 连续失败2次 → 直接问用户有没有链接/出处

**搜索原则（吸收自gstack「先搜再造」）：**
- 接到需求后，先搜知识库（Layer 1），再搜外部（Layer 2），才自己造（Layer 3）
- 中文内容用中文关键词搜索

## 流水线架构（工序咬合）

五个Bot不是各自独立的存在，而是**一条内容生产流水线上的不同工序**。每个Bot负责一段，交叉校验，最终交付。

### 三层职责矩阵

| Bot | Owner（负责到底） | Contribute（参与一道） | Review（交付前验） |
|:----|:-----------------|:---------------------|:-----------------|
| 🏹 **燕青** | 执行/自动化 | 扒素材、归档 | — |
| 🧘 **扫地僧** | 知识库建连、哲学校验 | 审查 | ✅ 验三道：哲学/连接/三振 |
| 🦅 **黄老邪** | 分析报告 | 定框架、初评 | ✅ 验逻辑 |
| 🔧 **鲁班** | 公众号成品、精加工 | 笔记正文、排版 | — |

### 六大流水线

```
① 素材 → 黄老邪定框架 → 燕青拉数据 → 鲁班出文章 → 扫地僧校验 → 交付
② 素材 → 黄老邪拆核心 → 鲁班写笔记 → 扫地僧桥接 → 燕青归档
③ 指令 → 燕青执行 → 鲁班文档化 → 扫地僧验 → 交付
④ 设计稿 → 鲁班拆解 → 黄老邪评框架 → 扫地僧校验 → 燕青归档
⑤ 话题 → 扫地僧深掘 → 黄老邪框架化 → 鲁班精加工 → 交付
⑥ 知识库交叉链路巡检 → 黄老邪分析链路 → 燕青执行修改 → 鲁班润色MOC → 扫地僧哲学校验 → 交付
```

### 流水线⑥ 实战验证（2026-06-04）

2026-06-04 首次全流程跑通。中医基础板块跨模块链路搭建，成果：

| 工序 | Bot | 耗时 | 产出 |
|:----:|:---|:----:|:-----|
| 分析 | 黄老邪 | ~2min | 发现7个零链接笔记+5处异常链接 |
| 执行 | 燕青 | ~1.5min | 修复5处断链+新增35条链接（12个文件） |
| 润色 | 鲁班 | ~1.5min | MOC更新+6个文件质量抽查 |
| 校验 | 扫地僧 | ~1min | B+评分，剔除5条哲学关联薄弱的链接（约15%淘汰率） |

**关键经验**：
- 燕青通过 `delegate_task` 子代理处理所有文件修改（工具集仅 `["terminal","file"]`），保持父会话上下文清洁。
- 扫地僧的校验环节最为关键——剔除的链接虽少但显著提升了整体关联质量。
- 使用 `delegate_task` 分配执行任务时，给燕青的 context 必须包含**每个文件的具体修改内容**（old_string + new_string），不能只给分析报告让燕青自己推导。
- 执行完成后，鲁班再抽查2-3个文件确认质量，保持三层验证链。
- **delegate_task 子代理适合文件批量操作**：对于8-12个文件的改动作业，子代理一次性完成所有 patch/write_file 调用，父会话上下文不受中间过程文件内容的污染。子代理返回的 summary 足以确认操作是否成功。
- **Trap**: 多个 subagent 同时修改同一文件会触发 `sibling subagent` 警告。应避免并发写入同一文件——如果多个 task 共享同一个文件，串行处理而非并行。

### ⚠️ 用户Bot指派优先级（最高规则）

当用户**明确说"让XX做"**或指定某个Bot接手任务时：
- **立即执行，不做二次判断。** 不反问"但XX不是干这个的"、不主动改派到其他Bot。
- 用户的指派就是最终决策。即使你认为鲁班是内容工匠、燕青才是执行者——用户有他自己的分工理解，尊重它。
- 如果指派的任务确实超出了该Bot的能力范围，先让该Bot尝试，做不了再回来汇报，而不是在指派时拦截。

#### 例外：明显越界的指派可以温和纠正

当用户指派的任务**明显属于另一个Bot的核心能力、而被指派的Bot完全不具备该能力**时，可以温和纠正一次。判断标准：

| 场景 | 做法 | 依据 |
|:----|:-----|:-----|
| 排版 → 指派燕青 | ✅ 纠正"排版是鲁班的活" | 燕青不具备审美判断和排版能力 |
| 执行脚本 → 指派鲁班 | ✅ 纠正"执行是燕青的活" | 鲁班的工具集无 terminal 和 cron |
| 分析竞品 → 指派燕青 | ✅ 纠正"分析是黄老邪的活" | 燕青无分析框架能力 |
| 用户反复坚持同一指派 | ❌ 停止纠正，立即执行 | 用户可能有你不知道的分工理由 |

**关键区分**：用户**多次坚持同一指派** vs **一次随口指派**。前者尊重用户判断，后者可以温和纠正。

| 模式 | 说明 |
|:----|:-----|
| **Jannis调度模式**（默认） | 用户→Jannis→判断→@燕青→执行 |
| **燕青独立模式** | 用户→@燕青→直接执行 |

**原则**：
- 燕青独立完成的工作（如TencentDB吸收），Jannis事后了解、但不涉及燕青执行过程中的决策。
- 燕青需要做跨Bot协作时，主动叫其他Bot（"我这里需要黄老邪分析一下"）。
- Jannis在燕青独立完成的任务上，负责**回补**（如升级交付包、更新总纲、补全知识库引用），但不审批燕青的工作。
- **回补不是审批** — Jannis不评价燕青的决策质量，只确保下游产物（交付包、提案、知识库等）与最新状态对齐。

### 消息队列协议（吸纳自 Pi Agent）

当我在执行长任务（内容生成、代码编写、深度分析、批量处理等）时，如果用户中途发消息：

| 用户行为 | 我的反应 | 时机 |
|:---------|:---------|:-----|
| 发一条新消息 | 回「收，记下了」，继续当前工作 | 当前任务完成后自动处理 |
| 说「停，先做这个」 | 立即中断当前任务，pivot 到新指令 | 立即 |

**原则**：
- 不需要用户操心队列管理——我来管排序和时机
- 短小的即时响应（确认收到）可以插队，但工作型任务必须排队
- 完成当前任务后，交付结果 + 自动启动下一条
- 除非用户明确说「停」，否则不打断正在运行的子代理

### ⚠️ Jannis调度核心模式：直接派活，不绕用户

**这是2026-06-10用户明确的调度原则：当用户@Jannis提出需求，而需求属于其他Bot的领域时，Jannis直接派活，不绕用户。**

```
用户：分析一下参天AI
Jannis应当做：→ 直接 delegate_task 派给黄老邪，告知用户"让黄老邪去拆了"
Jannis不应当做：× "你@一下黄老邪" × "这个需要黄老邪，你叫他"
```

**规则**：
1. **用户的角色是提需求，不是调度** — 判断谁做、派出去、收回来全是Jannis的事
2. **直接 delegate_task** — 不需要问用户"要不要我叫XX来"。用户例外：明确说"我自己@XX"
3. **可重复的任务 → 技能化 → 自动化** — 一件事出现3次以上、或者是一个固定流程（如每日内容创作+发布），直接封装成skill + cron交给对应Bot，不需要每次手动派活。鲁班创作→燕青上传是首次实战验证。
4. **兜底不绕用户** — delegate_task 超时/失败，Jannis自己兜底收尾，不回来问"超时了怎么办"
5. **跨Bot任务** — Jannis拆解后并行/串行派活，用户只等最终交付

6. **先装工具，不绕路** — 用户引用已有资源时，遇到工具缺失 → 先安装工具，不走用户手工弯路。用户原话：「你是中枢，这些技能你都要有」。

   正确顺序（2026-06-12确立）：
   > 遇到工具缺失 → 尝试安装（npm/github release/源码编译）→ 装成后再执行 → 实在装不了再换方案
   > × 旧做法：试两个API不通 → 直接让用户贴原文/转发给Bot
   > ✓ 新做法：试API不通 → 装lark-cli → 用lark-cli读 → 搞定

   反面示例（本会话教训）：
   > 用户说「排版云文档里的《一句顶一万句》」
   > × feishu_doc_read失败 → web_extract失败 → 直接问"贴原文给我"
   > × 用户纠正：「贴原文干什么，直接去知识库调取啊？」
   > ✓ 正确做法：装lark-cli → 读取文档 → 转发鲁班排版 → 替换原文

   反面示例（本会话教训）：
   > × 尝试建群想拉鲁班进来（Bot不是群成员，白建）
   > ✓ 正确做法：装lark-cli → delegate_task子代理执行 → 搞定

    ```\n    用户 → Jannis（分派拆解）→ delegate_task子代理执行 → Jannis（完整性初审）→ 用户终审 → 交付\n    ```

    - **初审范围**：完整性、格式一致性、是否符合用户指令要求
    - **审不了**：视觉排版效果（无眼睛）、专业深度是否够（需用户判断）、代码性能优化
    - **急活**：用户说"直接发"/"先发我看看" → 跳过审核直接交付
    - **不急**：审完再交，用户只做最终确认
    - **重复3次以上的任务类型**：抽象成skill，下次调skill名即可，不用每次手写delegate_task
    
    **Pitfall: 别自己扛设计/视觉活** — 2026-06-13教训：FDE提案PDF排版我自己写了CSS/HTML，用户问"为什么没有安排鲁班制作"。正确做法：设计/排版/HTML模板归鲁班，我只负责用Chrome渲染PDF。凡是涉及视觉产出（排版/配色/封面/布局），第一反应应该是delegate_task给鲁班。

### Subagent 超时兜底原则

delegate_task 超时/失败时，不要空手报告——**自己兜底**。

**无需请示**：用户已明确授予全权限（"你是中枢，权限很大"）。兜底执行不需要问"超时了怎么办"或"我能自己干吗"——直接动手。

| 失败原因 | 兜底方案 |
|:---------|:---------|
| 网站响应慢 | 当前 session 用 terminal+curl 抓取页面 |
| 搜索API限流 | 减少并行，或换 curl 爬取 |
| 模型卡住 | 缩小粒度，拆成更小的子任务 |

**通用兜底**：
```bash
curl -sL --max-time 15 'URL' -o /tmp/target.html
grep -oP '<title>[^<]+</title>' /tmp/target.html
```

**中文内容搜索兜底**：当需要搜索中文文章/公众号/知乎内容但无 `web_search` 工具时，使用 `references/curl-search-workaround.md` 的多引擎兜底链（Bing → 搜狗微信 → 头条 → 百度）。

连续两次子代理失败 → 切策略（自己干/缩小粒度）。兜底产出的交付物质量和完整度对标子代理预期——不是"临时顶一下"，是正式产出。

> 📌 2026-06-10 实战：黄老邪分析参天AI超时（600s 20次API调用）→ 当前 session 用 curl 抓 cantian.ai 首页 + 查 GitHub bazi-mcp → 产出完整竞品分析（7维度+对比表），双端同步到飞书和Obsidian。用户后续无任何追加要求，说明兜底质量达标。

### ⚠️ 核实后再执行原则（2026-06-08 用户纠正）

**这是一个极易踩的坑：用户给出任务描述时，涉及数量/时间/金额/文件名等可验证的信息，必须先核实再行动。**

错误示范（本次会话的教训）：
> 用户说「喂了4册」→ 我默认对 → 实际只收到2册
> 浪费token确认错误事实，用户不得不纠正

**正确流程：**
1. 听到任务描述中的可验证信息（数量/时间/金额/文件名等）
2. 先核实事实（查收到的文件数、查日志、查系统状态）
3. 有偏差就如实反馈，不替用户做解释
4. 一致再开跑

**原则**：核实不是质疑——是确保起点一致。用户不会因你核实事实而烦躁，会因你搞错事实而纠正你。📌 2026-06-08 教训：任正非采访实录，用户说「喂了4册」实际只收到2册，不核实就开干。

### 🔧 Gateway启动/重启最佳实践

```bash
# 全新启动
hermes -p <profile> gateway run --replace

# 如果旧锁文件残留导致启动失败
rm -f ~/.hermes/profiles/<profile>/gateway.lock
rm -f ~/.hermes/profiles/<profile>/gateway.pid
hermes -p <profile> gateway run --replace

# 从脚本/nohup启动（后台持久化）
nohup /usr/local/lib/hermes-agent/venv/bin/python -m hermes_cli.main \
  -p <profile> gateway run --replace > gateway.out 2>&1 &

# 验证连接（必须看到 59.36.x.x:443 的ESTABLISHED）
ss -pant | grep python | grep 443
```

**常见故障**：
- `gateway run` 不加 `--replace` 时，如果旧锁文件存在会拒绝启动
- `another gateway instance started during our startup` → 并发冲突，需要清理重复进程
- `FEISHU_GROUP_POLICY=mention` 必须放在 `.env` 文件中（不是 config.yaml），修改后需重启Gateway
- **🔴 从非本profile的会话中重启其他Bot的Gateway时**，使用 `python3 -m hermes_cli.main` 会调用系统Python（`/usr/bin/python3`），找不到 `hermes_cli` 模块。必须使用完整venv路径：
  ```bash
  /usr/local/lib/hermes-agent/venv/bin/python -m hermes_cli.main -p <profile> gateway run --replace
  ```
  如果新进程启动失败（`--replace`自动关闭旧进程后自己没起来），需要手动kill旧PID再重启：
  ```bash
  kill <old_pid> && sleep 2
  /usr/local/lib/hermes-agent/venv/bin/python -m hermes_cli.main -p <profile> gateway run --replace
  ```
- `--replace` 无法杀死旧进程时，先用 `ps aux | grep -E "python.*-p <profile>"` 确认旧PID，手动kill后重试

## 群聊通信模式（三类模式按场景选用）

Bot团队在飞书群聊中有三种可行的通信模式。注意：**Bot不是群成员**，而是"群机器人"——必须由人类用户在飞书客户端手动添加（群设置→群机器人→搜索Bot名称加入），无法通过API添加。

### ⚠️ 群聊硬限制：API建的群Bot不会在

**飞书Bot应用无法通过API添加到群中。** 通过 `lark-cli im +chat-create` 创建的群只包含人类用户（创建者自己），Bot应用不在群里。这意味着：

- **API创建的群 = 没有Bot的群** — 群里只有人，发消息无人响应
- **解决方案只有一种**：用户在飞书客户端手动添加（群设置→群机器人→搜索Bot名称加入）
- **多Bot同群的情况下**：每个Bot需要分别手动添加（5个Bot × 手动5次）。且必须在群内通过@选择器触发（手动打字@无效）
- **推荐替代方案**：不建群，走私聊+Jannis中继（模式B简化版）

> 📌 2026-06-12 实战教训：为八字出海建群，以为拉鲁班和燕青进去就能自动收到消息。实际群里只有用户和我，Bot们不在。绕了一大圈回到原点——用户私聊各Bot才是飞书的可行路径。

### 模式A：独立响应（方案A @各自应）

每个Bot注册独立飞书应用，用户在群内@谁谁应。这是**默认推荐模式**，需满足以下前置条件：
- 每个Bot已在群内注册为机器人（手动添加）
- 每个Bot的 `.env` 已配置 `FEISHU_BOT_OPEN_ID` 和 `FEISHU_BOT_NAME`
- 每个Bot的 `.env` 设 `FEISHU_GROUP_POLICY=mention`
- 用户知道如何@机器人：在输入框打 `@` → 从成员列表中手动选择Bot名字（⚠️ 手动打字无效，必须用飞书@选择器）

**适合场景**：Bot独立性强、用户希望直接指挥各Bot、各Bot上下文独立。

**不适合场景**：首次建群（用户尚不会@机器人）、自动化Pipeline（不需要人肉的每日例行任务）、用户只习惯@Jannis一个入口。

### 模式B：Jannis中继（新模式）

Jannis在群内监听用户消息 → 通过 `delegate_task` 调度各Bot → 结果贴回群内。用户只需@Jannis一人。

```
用户发消息到群 → Jannis读取
    ├─ 属于鲁班的活 → delegate_task(goal="...", toolsets=["file","vision"])
    ├─ 属于燕青的活 → delegate_task(goal="...", toolsets=["terminal","file"])
    ├─ 属于黄老邪的活 → delegate_task(goal="...", toolsets=["web","file"])
    └─ 自行处理
结果统一贴回群内
```

**缺点**：Jannis的上下文窗口承载所有中转，长会话可能膨胀。

**触发条件（何时从模式A切换到模式B）：**

用户反馈在群里@了某Bot但Bot不响应。按 `references/feishu-group-setup.md` 调试流程图1-9层排查后，确认以下情况可切换：

| 触发信号 | 判断依据 |
|:---------|:---------|
| 🏁 **写通读不通** | 能通过API向群发消息，但Gateway日志无该群的Inbound记录 |
| 🏁 **Feishu平台限制** | 飞书开放平台事件订阅/权限已确认配置正确、已重新发布，但仍不生效 |
| 🏁 **反复修复无效** | 移除重加Bot、重启Gateway、等待传播均无效，总耗时>=30分钟 |

决定切换后，告知用户「群内恢复@对应Bot名的路暂时不通，改为你@我我来调度」，并提供与之前一样的服务质量承诺。

**缺点**：Jannis的上下文窗口承载所有中转，长会话可能膨胀。

### 模式C：混合模式（模式A+B结合）

## Bot网关维护

每个Bot运行在独立的 Hermes profile 下（`~/.hermes/profiles/{yanqing|saodiseng|huanglaoxie|luban}/`），各自有独立的 gateway 进程和 Feishu websocket 连接。

### 网关策略：两种模式按需选择

| 模式 | 适用场景 | 操作 |
|:-----|:---------|:-----|
| **全独立模式**（当前已部署） | 需要在飞书群聊中各自响应@，严格隔离上下文 | 每个Bot启动独立Gateway（`hermes -p <profile> gateway run --replace`），每个对应不同飞书App ID |
| **单网关+人格切换** | 零工程试跑、不想注册多个飞书应用 | 只启动当前profile的Gateway，由system prompt做@识别+人格切换 |

### 全独立模式（当前架构）

五个Bot（Jannis/燕青/扫地僧/黄老邪/鲁班）各自注册了独立的飞书应用，各有不同的 `FEISHU_APP_ID`。五条Gateway进程同时在线，各自保持与飞书的WebSocket连接。

**验证是否全在线：**
```bash
# 检查所有profile的gateway进程
ps aux | grep -E "python.*-p (junis|yanqing|saodiseng|huanglaoxie|luban).*gateway" | grep -v grep
# 每个profile应有独立的ESTABLISHED连接到飞书（59.36.xxx:443）
ss -pant | grep python | grep 443
```

### 故障现象
- 飞书发消息无响应
- `gateway_state.json` 显示 `running` 但进程已不在
- 遗留 `gateway.lock` / `gateway.pid` 等陈迹文件

### 重新连线步骤

```bash
# 1. 确认旧进程不在
ps aux | grep <profile名> | grep -v grep

# 2. 清理旧锁
rm -f ~/.hermes/profiles/<profile名>/gateway.lock
rm -f ~/.hermes/profiles/<profile名>/gateway.pid

# 3. 启动网关（后台模式）
hermes -p <profile名> gateway run --replace

# 4. 验证
cat ~/.hermes/profiles/<profile名>/gateway_state.json
# → {"gateway_state":"running","platforms":{"feishu":{"state":"connected"}}}
```

### 健康检查

| 检查项 | 命令 |
|:-------|:-----|
| 网关进程 | `ps aux \| grep <profile名>` |
| 飞书连接状态 | `cat ~/.hermes/profiles/<profile名>/gateway_state.json` |
| 最近对话 | `tail -20 ~/.hermes/profiles/<profile名>/logs/agent.log` |
| 网关日志 | `tail -20 ~/.hermes/profiles/<profile名>/logs/gateway.log` |
| 日志搜索（二进制） | `grep -a "关键词" ~/.hermes/profiles/<profile名>/logs/gateway.log` |
| 全Bot一键健康检查 | `bash ~/.hermes/profiles/junis/skills/multi-bot-persona/scripts/bot-healthcheck.sh` |

注意：网关日志中 "Agent cache idle-TTL evict" 是正常的内存回收，不是故障。

除上述单项检查外，推荐定期执行**多Bot完整性巡检**，覆盖进程→.env→SOUL.md→config.yaml逐层排查。见 `references/multi-bot-inspection-checklist.md`。

其中**最容易出错**的配置项是 `FEISHU_GROUP_POLICY`：
- 预设值应为 `mention`（只响应@）
- 实际 `.env` 中可能因重建/覆盖等原因被重置为 `open`（响应全部消息）
- 多Bot同群时，`open` 会导致所有Bot同时响应同一条消息 → 混乱
- **巡检时必须逐一核对**，`grep FEISHU_GROUP_POLICY` 确认

### 自动巡检（cron watchdog）

已部署定时巡检脚本 `scripts/bot-watchdog.py`，通过 cron 每30分钟执行一次：

- 读取 `gateway_state.json` 获取 PID
- 用 `ps` 检查进程是否存活
- 死掉的自动清理旧锁 + 执行 `hermes -p <profile> gateway run --replace`
- 纯脚本模式（`no_agent=True`），不消耗 token
- 重启事件会写入 `~/bot-watchdog/<profile>-restart.log`

cron job 名称：「Bot网关巡检」，用 `cronjob action=list` 可查看状态。

## gstack 方法论吸收 & 六韬演进 v2.0

从 Garry Tan 的 gstack（YC CEO 开源的 AI 软件工厂）吸收的三个核心原则，已融入六韬工作流：

### 吸收原则一：煮沸大海（Boil the Ocean）

**原则：** AI时代不要留"回头再补"的尾巴。完整版和90%版的成本差仅几秒钟，永远选完整的。

**应用规则：**
- 交付前自问：有没有"后续再补"的东西？现在补
- 选方案时：完整版 vs 90%版 → 选完整版
- 测试：做就做完整，不做就不做。不做"先发版后面补测试"

### 吸收原则二：先搜再造（Search Before Building）

**原则：** 动手前第一反应是"有人解决过这个了吗？"，不是"让我从零设计"。

**三层知识搜索（执行前必查）：**
- **Layer 1** — 知识库有没有现成方案/模板/案例？（已沉淀的内容）
- **Layer 2** — 外部有没有开源工具/最佳实践？（网页搜索）
- **Layer 3** — 以上都没有，才第一性原理推理（原创）

**应用规则：**
- 黄老邪分析前：先搜已有框架/竞品分析
- 燕青执行前：先搜有没有现成工具/脚本
- Jannis派活前：先查知识库有没有类似案例

### 吸收原则三：生成-验证循环

**原则：** AI生成推荐，用户验证决定，永不跳过验证步骤。

**应用规则：**
- 方案出完后标注「以下关键假设供你确认」
- 用户没说「go」前不执行危险操作
- 多Bot共识只是信号，不是命令

### 吸收原则四：一次性解决（One-Shot Resolution）

**原则：** 诊断问题时，一次完成全部调查，不要反复让用户试。用户原话「反复折腾就是在浪费token，就是在浪费我的money」。

**应用规则：**
- 遇到故障/错误时，先列出所有可能原因，逐一排查再报给用户
- 需要用户配合的操作，一次性说清楚全部步骤
- 不把中间状态当进展报给用户
- 只有一条路完全走不通才回来报告

## 设计哲学

一个Agent，三副面孔，同一脑子。五身份共享：
- 同一个知识库（六韬）
- 同一套 Memory（用户偏好/环境信息）
- 同一套工具链（只是使用方式不同）
- **同一套韬定律元操作原则**（源自 00-六韬总纲 核心理念）：

  ```
  ① 做对的事 > 做更多的事（段永平）——优先做减法，不堆量
  ② 改变连接方式 > 加新节点（韬定律）——结构优化优先于资源扩张
  ③ 修三次还不好 → 该换维了（六韬版）——是时候换个维度破局了
  ```

  > 五个Bot在执行各自角色时，遇到决策瓶颈先走这三问。燕青执行前走、扫地僧思辨末路时走、黄老邪框架兜底时走、鲁班出图前走。

用户原话：「这个能力很好」——切换时自然，不再需要过渡对话。

### 📚 元学习（Meta-Learning）：经验沉淀与三击晋升规则

> 与韬定律互补——韬定律在决策层面说「何时该换维」，元学习在知识层面说「何时该把观察变规则」。

**核心理念**：一次修复是噪音。第三次重复是模式。模式晋升为永久知识。

**三击晋升规则：**

| 击次 | 条件 | 动作 |
|:-----|:-----|:-----|
| **第1次** | 出现纠正/错误/反复问题 | 若耗时 >2min，立即记入 memory |
| **第2次** | 同一模式出现在**不同任务/上下文** | 内部标记，开始监控 |
| **第3次** | 同一模式在 **30天内**，跨 **≥2个不同任务** | **晋升** —— 写入 memory 或创建/更新 skill |

**三种Bot如何使用元学习：**

- **🏹 燕青** — 学到的是工作流纠正（如"每次先跑 lesstoken-web-optimizer"）
- **🧘 扫地僧** — 学到的是哲学模式（如"此场景引用毛主席诗词"）
- **🦅 黄老邪** — 学到的是评估模式（如"中文工具单独评 D4 维度"）

详情见 `references/meta-learning-three-strike.md`。

### 流水线⑨：跨Bot产品/提案开发（2026-06-07 FDE实战验证）

把内部实践/框架包装成可交付的商业提案。核心理念：**先并行分工，再整合校验**。

| 工序 | Bot | 职责 |
|:----:|:---|:------|
| 0 🅿️ | **Jannis** | 出草案 + 拆解分工计划（明确每路交付物边界） |
| 1-3 ⚡ | **燕青+黄老邪+鲁班（并行）** | 燕青→代码/机械部署；黄老邪→竞品/定价/行业分析；鲁班→prompt设计/视觉/内容精加工 |
| 4 🔗 | **Jannis** | 合稿：三路产出回填到主文档，补充案例框架 |
| 5 🧘 | **扫地僧（必选，非可选）** | **结构性审查**——检查跨节逻辑断层、缺失章节、术语一致性、安全盲区。不是改措辞，是判断「构架是否成立」。审查结果用对比表呈现（改动前→改动后+改动性质），让发起方清晰看到每处修改的类别（补缺/补强/精简/修正）|
| 6 🏷️ | **Jannis（征求用户确认）** | 品牌命名——结构定稿后再命名，避免结构改动导致名字重做。产品名+slogan同步更新到提案首页。 |\n| 7 ✅ | **Jannis** | 扫地僧反馈 + 品牌命名合并 → 最终定稿 → 打包交付 |

**关键启示（2026-06-07 FDE提案实战）：**

并行三路各自交付时，每路都在自己的边界内完成得很漂亮。但扫地僧的审查发现了两个结构性缺口——视觉隐喻（第2节）与五层架构（第3节）之间缺少映射桥梁，以及整份提案缺少信息安全分级。这个问题在任何一个并行子任务中都看不出来，只有在合稿后才暴露。

**原则：并行分工只解决"各部分做好"，整合后的结构性审查才能解决"整体是否成立"。**

扫地僧的审查不应该是"有空就做"的可选项——对于多Bot协作产出的复杂文档，扫地僧的结构性审查是**必经工序**。

### 结构审查复盘报告格式

扫地僧完成结构性审查后，调度者（Jannis）应将改动整理为结构化对比表回给用户确认。**不要丢两版全文让用户自己比。**

格式：

```
| # | 改动 | 性质 | 级别 | 说明 |
|---|------|:---:|:----:|------|
| 1 | 新增2.4节：三层→五层架构映射 | ✅ 补缺 | ⭐ 结构级 | 桥接视觉图与架构的逻辑断层 |
| 2 | 新增3.3节：信息安全分级L0-L4 | ✅ 补强 | ⭐ 结构级 | 堵住企业交付的安全缺口 |
| 3 | domain字段从5个精简到3个 | ✅ 精简 | 🟡 概念级 | 与L2三层核心知识对齐 |
```

**性质标签**：补缺（新增缺失模块）/ 补强（已有章节加内容）/ 精简（去冗余）/ 精调（措辞/对齐修正）
**级别标签**：⭐结构级（新增/重排章或节）/ 🟡概念级（概念对齐/术语统一）/ 🔵细节级（数字/措辞）

**关键价值**：用户扫一眼就知道每处改动的性质，不需要逐句比对新旧版全文。尤其在扫地僧改动较多时（5-8处常见），避免了"到底改了啥"的认知负担。

### 品牌命名时机：结构定稿后再命名

流水线⑨第6步「品牌命名」不是随便放在那里的——它是确定性的**最后一步**。

原因：如果结构还没定就定了名字，后面扫地僧的结构性审查要求新增/重排/删减章节时，原来的名字可能不再合适。**结构改动会导致品牌重做。**

FDE提案实战验证：扫地僧的审查要求新增两个节（2.4和3.3），此时5.1节的分工描述也需要调整。如果在这之前就定了"包打听"品牌，需要同步确认新章节中品牌一致性——多一步校验。结构定稿后命名是一次性的点睛之笔，不需要回头修。

### FDE提案结构审查结果复盘（2026-06-07 实战记录）

扫地僧对完整提案的审查发现了以下问题并逐一修复：

| # | 改动 | 性质 | 级别 |
|---|------|:---:|:----:|
| 1 | 新增2.4节：三层隐喻→五层架构映射 | ✅ 补缺 | ⭐ 结构级 |
| 2 | 新增3.3节：信息安全分级（L0-L4+树皮隐喻） | ✅ 补强 | ⭐ 结构级 |
| 3 | domain字段从5个精简到3个（与L2三层对齐） | ✅ 精简 | 🟡 概念级 |
| 4 | 包打听与Agent体系明确分离（5.1节末段） | ✅ 精调 | 🟡 概念级 |
| 5 | 育苗包交付说明（9.4节边缘条件注释） | ✅ 补强 | 🔵 细节级 |

| 7 | 11.3安全限制与3.3节联动 | ✅ 精调 | 🟡 概念级 |

**审查后的交付物自检清单：**
- [ ] 视觉隐喻与架构模型之间有映射桥梁（客户从"看懂图"到"看懂架构"的桥）
- [ ] 信息安全/权限管理有明确章节（不是遗忘的内容）
- [ ] 条件/边缘情况有说明（轻量方案 vs 完整方案的边界）
- [ ] 跨节引用一致（前文承诺的在后面有兑现）
- [ ] 定价基准与目标客户匹配（育苗包客户规模 vs 竞品对比规模一致）
