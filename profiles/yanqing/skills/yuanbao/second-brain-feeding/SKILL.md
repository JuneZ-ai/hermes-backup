## 三、喂料流程

### Step 0：获取内容

| 来源 | 操作 |
|------|------|
| 飞书发的 PDF/图片 | 读取本地缓存文件 |
| 用户粘贴的文本 | 直接使用 |
| 飞书文档链接 (`feishu.cn/docx/...`) | **走 Feishu API 读取**（见下方「飞书文档API读取」）——注意 `feishu_doc_read` 工具仅在飞书评论上下文中可用，DM/群聊中必须走 API |
| MinerU/OCR 导出的 markdown 文本 | **清洗+格式化**（见下方「MinerU/OCR 文本处理」） |
| 飞书云盘 `file` 文件 (`my.feishu.cn/file/...`) | **lark-cli 下载**（见 [[references/feishu-drive-file-download.md]]） |
| GitHub 链接 | `curl -sL raw.githubusercontent.com/...` |
| **GitHub 项目描述（非链接）** | 见下方「GitHub 开源项目代码库调研专项」 |
| Google/Gemini 等被墙链接 | **无法直接访问**，告诉用户贴过来 |
### 通用网页抓取策略排名

| 策略 | 成功率 | 适用场景 | 注意 |
|:----|:------|:--------|:-----|
| **curl + User-Agent + max-time 15s** | ~70% | 大多数网站 | 默认首选 |
| **python3 urllib + headers** | ~65% | curl被墙时备用 | 超时30s |
| **python3 requests** | ~50% | 某些特定API | 可能因网络环境hang |
| **delegate_task + web工具集** | ~80% | 综合搜索+内容提取 | 注意AI幻觉风险 |
| **浏览器工具** | ~90% | JS渲染页面 | 最可靠但最慢 |

**⚠️ WAF 防护网站处理：** 遇到阿里云WAF/Akamai/CDN防护的页面（如 v2su.com 等），直接标记为"WAF拦截不可访问"，不要反复重试。改用子任务搜索替代来源。

**网站超时分级处理：**
```python
timeout 15 curl -sL <url>       # 15秒内返回 → 读取内容
timeout 15 curl -sL <url>       # 超时 → 换python urllib重试（30s超时）
                                # 仍不可达 → 标记"连接失败"并报告
```

#### 飞书文档 API 读取（当 `feishu_doc_read` 不可用时）

用户通过 DM/群聊发送 feishu.cn/docx/ 链接时，`feishu_doc_read` 工具只在飞书评论上下文中可用。DM 中须走 API：

**方式 A（推荐）：用已有脚本获取 token + curl 读文档**
```bash
# 1. 用已有脚本获取 tenant_access_token
TOKEN=$(/usr/bin/python3.14 /home/hermes/.hermes/scripts/get_token.py)

# 2. 读取文档 raw_content
curl -s -m 15 -H "Authorization: Bearer $TOKEN" \
  'https://open.feishu.cn/open-apis/docx/v1/documents/{doc_token}/raw_content' \
  -o /tmp/feishu_doc.json

# 3. 用 read_file 工具读 /tmp/feishu_doc.json 提取 content 字段
```

**方式 B（行内 curl）：**
```bash
TOKEN=$(curl -s -m 10 -X POST \
  'https://open.feishu.cn/open-apis/auth/v3/tenant_access_token/internal' \
  -H 'Content-Type: application/json' \
  -d '{"app_id":"cli_aa869b8d6b3c5cc3","app_secret":"qSk4tz1A15aWo5K4eFw2lbhibgMAVaHN"}' | \
  python3 -c "import sys,json; print(json.load(sys.stdin)['tenant_access_token'])")

curl -s -m 15 -H "Authorization: Bearer $TOKEN" \
  'https://open.feishu.cn/open-apis/docx/v1/documents/{doc_token}/raw_content' \
  -o /tmp/feishu_doc.json
```

**方式 C（Python stdlib — 当 curl 也不可用时备用）：**
```python
import urllib.request, json
token = json.loads(urllib.request.urlopen(urllib.request.Request(
    'https://open.feishu.cn/open-apis/auth/v3/tenant_access_token/internal',
    data=json.dumps({'app_id':'cli_aa869b8d6b3c5cc3','app_secret':'qSk4tz1A15aWo5K4eFw2lbhibgMAVaHN'}).encode(),
    headers={'Content-Type':'application/json'}
)).read())['tenant_access_token']

req = urllib.request.Request(
    'https://open.feishu.cn/open-apis/docx/v1/documents/{doc_token}/raw_content',
    headers={'Authorization': f'Bearer {token}'}
)
content = json.loads(urllib.request.urlopen(req).read())['data']['content']
```

> ⚠️ Python `requests` 可能超时（网络环境原因），优先用 `curl` 而非 Python。`curl` + `timeout` 前缀（`timeout 12 curl -s -m 10 ...`）可防止挂起。
> 详细流程与故障排查见 [[references/feishu-doc-api-reading.md]]

**doc_token 提取规则：** 从 `https://www.feishu.cn/docx/{doc_token}` 或 `https://xxx.feishu.cn/docx/{doc_token}` 中提取 `/docx/` 后的字符串（不包含查询参数）。

**⚠️ 拿到 raw_content 后不要直接 dump 到 .md！** raw_content 是纯文本，需要手动转换为 Obsidian 格式。转换方法取决于内容类型（结构化笔记/纯文章/表格密集文档），详见 `[[references/feishu-doc-api-reading.md#Post-Processing：raw_content-→-Obsidian-Markdown]]`。

### Step 1：清洗内容（重要！）

**尤其对于 AI 生成的文本（Gemini/ChatGPT 等）：**

🔴 必须删除：
- 对话式追问（"您目前是倾向于…？" 这类引导问题）
- 来源 AI 的交互式 UI 文本
- **明确错误/误导的内容**（例如 Gemini 把 Hermes Agent 和 Cosmos IBC Relayer 混为一谈）
- 冗余重复的段落

🟢 保留：
- 定义、分类、机制等硬核知识
- 可操作的框架/模型
- 有参考价值的案例和数据

#### MinerU/OCR 文本清洗要点

当内容来自 MinerU 等 OCR 工具导出的 markdown 时，需额外清洗：

1. **HTML 表格提取**：MinerU 常将表格渲染为 `<table>` HTML 标签。检查文件中是否有 `<table>...</table>` 块。若有，将其内容提取为 markdown 表格或段落文本
2. **合并行分裂**：OCR 常把相邻行连在一起（如七律的两句拼成一行）。检查每首诗的句数和结构，在正确的位置断行
3. **全角字符归一**：OCR 输出常使用全角引号、逗号等 variant 字符。不影响阅读的保留，影响格式化（如 `、` 与 `，` 混用）的归一化
4. **结构重建**：OCR 输出的诗/词等结构化文本常常丢失原本的段落/章节结构。参照标题和日期标记，按「标题（年代+诗体）」→「日期」→「正文」三段式重排
5. **附注分离**：诗文中的长序言/注释用 `> blockquote` 包裹，与正文分离

**详细指南见 `[[references/mineru-ocr-processing.md]]`**

## 材料筛选三原则

> **在进入任何清洗/入库动作前，先用三把尺子量来源。不满足的，不入库。不满足三条但满足"来源权威"一条的补充型数据（薪资/案例数据），合并入已有笔记。全不满足的，跳过。**

| 标准 | 要求 | 判断方法 |
|:----|:-----|:---------|
| **① 来源权威** | 业界公认的专业网站/平台 | ✅ YC / TechCrunch / Wired / Bloomberg / Palantir官方博客 / OpenAI官方 / Anthropic官方 / 头部AI公司工程博客 / 权威学术会议/期刊 |
| **② 人物顶尖** | 行业内公认的顶尖人物 | 创始人/CEO/首席科学家/领域开创者（如黄仁勋/Sam Altman/Bob McGrew/Colin Jarvis/Andrew Ng）。注意区分"转述"和"直接观点"——转述不满足此条 |
| **③ 前瞻洞察** | 有未来判断/趋势预测，不只是回顾 | 含"预测""看好/看空""下一步""趋势""3-5年"等未来指向信号 |

**例外：补充型数据更新**（如薪资数据、新案例数据）可只满足来源权威一条，直接合并入已有笔记，不单独建笔记。

评估框架可信度的关键提醒： 当一个框架/方法论看起来是"自建"的，先问——这背后有没有顶级人物公开验证过？τ定律是自建框架，但黄仁勋15年CUDA布局就是最强实证。别因为没有外部链接就否定其价值。

#### 合并入库 vs 新建笔记的决策

并非所有内容都要创建独立笔记。遇到以下情况优先选择**合并入已有笔记**而非新建：

| 场景 | 操作 | 示例 |
|:----|:-----|:-----|
| **补充行业数据**：已有方法论笔记，新文章提供更新数据/案例 | 在方法论笔记末尾新增「行业前沿动态」小节，不单独建笔记 | 53AI FDE文章 → 整合到 ZN-26 §6 |
| **更新已有知识**：已有笔记内容部分过时 | 替换/增补对应小节，更新 timestamp | 行业薪资数据更新 → 替换 ZN-26 §6.2 |
| **补充同一主题的深度分析**：新文章角度不同但主题一致 | 在已有笔记中新增章节，保留来源 | 不同FDE文章的不同角度 → 各自成节，统一在 ZN-26 |

**何时必须新建独立笔记：** 新主题无对应笔记、全新方法论/框架、完整书籍/文档初次入库。

### 2B：时事热点快速入库

当用户分享**热点视频/科技突破/时事解读**（如华为韬定律视频）时：

**核心原则：** 不是原文搬运，而是**提炼本质 + 六韬映射**。因为用户分享时已经做了选择——他看到的是这件事与知识库的内在联系。

**流程：**

1. **聆听/阅读**：理解热点本质（3句话以内）
2. **找到知识库连接点**：
   - 它对应哪个模块？→ 智脑（方法论）、道史（历史鉴证）、易哲（智慧洞见）
   - 它和现有哪个笔记共振？→ 如韬定律和FDE方法论都讲"流程重构替代堆料"
3. **创建结构化笔记（3000字内）**：
   - frontmatter（含 tags/aliases）
   - 一句话核心洞察
   - 电梯比喻（用用户能懂的语言解释）
   - 本质分析（从力大飞砖到精细运营）
   - 双关/巧合（如"韬"≠六韬但形成回响）
   - 关联阅读（指向已有笔记）
4. **用 ZN 编号（智脑）或对应模块编号命名**
5. **更新总纲 + 搭建日志**

**示例：华为韬定律入库** → `六韬智脑/ZN-32-华为韬定律·电梯分段法.md`

```
核心洞察：在物理极限面前，用流程重构替代堆料
六韬映射：与FDE方法论同构——都是"不加资源，改流程"
长度：~2800字，含电梯比喻+本质分析+双关注释+关联阅读
```

当用户分享的是 **ta自己的分析/框架/战略思考**（而非外部文章/PDF/链接），处理方式与外部内容完全不同：

**核心原则：** 用户自己的思考已经过提炼，不需要清洗，但需要结构化+体系化映射。

**流程：**

1. **确认框架核心**：用户想表达的核心观点是什么？提炼为3-5句话
2. **定位知识库位置**：判断属于哪个模块（通常是智脑/道史）
3. **创建「方法论」笔记**（X-框架名，含框架本身）
   - 保留用户原始框架结构（行业数据、能力矩阵、趋势分析等）
   - 补充六韬体系映射（见下方）
   - 补充实战场景/避坑清单
4. **创建「行动手册」笔记**（X+1-框架名行动手册，独立编号）
   - 方法论姊妹篇
   - 包含：Stage-by-stage Playbook / Checklist / 快速启动模板 / 工具引用
   - 让框架可落地、可复用
5. **前后关联**：方法论笔记末尾列出关联阅读，行动手册开头指向方法论笔记

**六韬映射模板（适用于战略/方法论类框架）：**

```
六韬[框架名]
│
├── ☯️ 太极双螺旋 → 元认知/时机判断（什么时候推/什么时候收）
├── 🧠 六韬智脑 → 战略决策（选什么/怎么选）
├── 🔮 六韬易哲 → 天机预测（风险评估/时机把握）
├── 📚 六韬道史 → 历史鉴证（历史案例对照）
└── ⚡ 决断之桥 → 贯通战略与执行
```

**并行优化**：方法论和行动手册两篇笔记可同时创建（互不依赖），提高效率。

### Step 2.5：体系贯通（跨笔记连接）

> 新内容可能与知识库中**已有的多篇笔记**形成体系连接。发现这种连接后，主动建立跨模块引用体系。

**什么时候做：** 新笔记创建后、更新索引前。先反思：这篇笔记和已有的哪些笔记"共振"？

### Step 5：同步飞书 Base（三表同步 + 自动留痕）

**留痕纪律（用户强调）：** 「即便没有动作也要留下留痕的记录」——即使今天没有任何喂料/入库操作，也要在每日记录和收藏随想录中写入一条空留痕，说明「今日无重大动作」。不留空白天。

**时序陷阱：日事日毕意味着手动兜底，不是等 cron。**

**规则：每次更新搭建日志后，看一眼时间。**
- 如果还没到 21:00 ✅ → cron 今晚会处理
- 如果已经过了 21:00 🔴 → **立刻手动同步，不等明天**

**不用等用户问。** 这是「日事日毕」的题中应有之义——今天更新的日志，今天进飞书 Base。

### 4.5 PDF 入库专项

> 用户发来 PDF 格式的书籍时（无 epub 可用），使用 PyMuPDF 提取文本。

#### 工具

- Python 库：`pymupdf`（fitz），已安装到 `/usr/bin/python3.14` 系统级
- 用法：`fitz.open('book.pdf')` → `doc[i].get_text()` 逐页提取
- 不支持 OCR（只提取内嵌可选文本）
- 参考：`[[references/pdf-to-markdown.md]]`、[[references/pdf-ocr-workflow.md]]（扫描件OCR策略，原 `pdf-processing` skill 已吸收）

#### 存储策略：MD 优先，PDF 不进 Vault

> **用户明确要求：PDF 转 MD 后入库，PDF 本身不放入 Obsidian vault（占用磁盘空间）。**

- 源 PDF 保留在 D 盘原始位置（`/mnt/d/...`），作为备份
- Vault 中只存 `.md` 文件（原文提取版或蒸馏版）
- 在 MD 文件的 frontmatter 或末尾标注 `来源PDF: /mnt/d/路径`，便于日后溯源

#### 内容类型标记（frontmatter）

每本书入库时，在 frontmatter 中标记内容类型，便于后续盘点：

```yaml
---
title: 三命通会
type: original-fulltext    # 原文完整版（PDF文字层直接提取，未删减）
# 或
type: distilled            # 蒸馏精华（框架笔记/读书笔记/摘要）
# 或  
type: original-extracted   # 原文提取版（PDF转MD，可能因OCR质量有缺失）
---
```

**判定规则：**
- PDF 文字层完整提取 → `original-fulltext`
- 扫描件 OCR 提取 → `original-extracted`（标注 OCR 质量）
- 框架笔记/读书笔记/摘要 → `distilled`
- 小说全文（epub/mobi 转 MD）→ `original-fulltext`

#### 处理流程

1. **检查 PDF 是否有文字层**：先用 `pymupdf` 提取前3页，如果字数 > 0 则有文字层
   - 有文字层 → 直接提取全文（fast path）
   - 无文字层 → 走 OCR 流程（见 4.5.1）
2. **提取文本**：逐页 `get_text()`，合并为完整字符串
3. **章节检测**：扫描 `第X章` / `上部` / `下部` / `前言` 关键词
   - 注意：PDF 的换行位置可能导致章节标题跨页中断，检测失败不必强求
   - 文本完整即可，章节标记作为二级优化
4. **构建 Markdown**：YAML frontmatter（含 type 标记）+ 元数据表 + 全书结构描述 + 正文
5. **双册结构**：上部+下部的作品（如《一句顶一万句》），用 `## 上部` / `## 下部` 分隔，不做精细 H 标签
6. **入库**：`六韬易哲/文史哲/<书名>-<作者>.md`
7. **索引更新**：六韬总纲的文史哲树 + 使用场景速查

#### 4.5.1 扫描件 OCR 流程（无文字层 PDF）

当 PDF 是扫描件（图片式）时：

1. **转图片**：用 pymupdf 把每页转成 PNG（`page.get_pixmap(dpi=200)`）
2. **批量 OCR**：
   - 方案 A：RapidOCR（ONNX Runtime，本地，无 API 成本）— 优先
   - 方案 B：Vision API（如当前环境可用的 vision_analyze）— 逐页，成本高，仅用于小文件或关键页面
3. **输出**：合并 OCR 文本为 MD，标记 `type: original-extracted` + `ocr_quality: low/medium/high`

**RapidOCR 快速路径（已验证）：**
```bash
# 安装
pip install rapidocr-onnxruntime -q

# 批量 OCR
python3 -c "
from rapidocr_onnxruntime import RapidOCR
ocr = RapidOCR()
for img in ['page_001.png', ...]:
    result, elapse = ocr(img)
    print('\n'.join([line[1] for line in result]))
"
```

**⚠️ 大文件 OCR 成本控制：**
- 427 页 × vision API ≈ 高昂成本 → 优先本地 OCR
- 如果本地 OCR 不可用且文件 > 100 页，**暂停并告知用户**，不要硬上 vision。保留转换好的 PNG 图片在 `/tmp/`，标注 OCR 状态供续传
- 先提取前 20 页（封面/目录/章节开头）构建目录框架，再决定是否全量 OCR

**暂停/恢复机制：** 当用户要求暂停 OCR（如"暂停千里命稿的处理"）：
1. 保留 `/tmp/<bookname>/` 下已转换的全部 PNG 图片
2. 在 vault 中存入标注 `待OCR` 的 placeholder MD（如 `千里命稿_待OCR.md`）
3. MD 文件头标注 `进度: 已转N页PNG` + `状态: 待OCR`，便于后续恢复
4. 后续恢复时：检查 PNG 数量确认完整性 → 继续 OCR 流程

#### 4.5.2 大型合集的拆分策略

当遇到多本书合订在一个 PDF 中的情况（如《文史哲入门三部曲》44MB/2087页）：

| 情况 | 处理方式 |
|:----|:---------|
| 有明确目录（TOC）且每本书起始页可定位 | 拆分为独立 MD 文件，分别入库 |
| 连续排版无明确分界 | **不拆**，保留蒸馏笔记即可。源 PDF 留 D 盘，vault 不放完整原文 |
| 用户明确要求拆分 | 用章节标题关键词搜索定位分页点 |

**判断标准：如果拆分定位成本 > 价值，不拆。** 蒸馏笔记已覆盖核心内容，源 PDF 可追溯。

### 4.6 多源深度研究（Multi-Source Deep Research）

> 当用户在短时间（同一会话/同一天）内连续发送同一主题的多个来源（多篇文章、PDF、播客逐字稿），应按"研究专题"而非"独立喂料"处理。

**触发条件：** 同一主题连续3+来源 / 来源类型多样（网页+PDF+访谈） / 用户带有明确意图（"继续学习补充"）

**处理流程：**
1. **逐个读取，不批处理**：每个来源都要读取完整内容
2. **识别"增补"还是"新建"**：补充数据→合并入已有笔记；全新视角→新建独立笔记
3. **跨来源合成**：所有来源处理完后做"对照分析"——同一主题不同来源的观点差异/互补
4. **双向交叉引用**：所有相关笔记互加wiki链接
5. **更新总纲贯通表**：在方法论笔记的"体系贯通"小节中列出所有新笔记及其关系

**合成最佳实践：**
- 提炼**增量价值**：每个新来源提供了前面没有的什么
- 对照表优先于独立总结：A说X，B说Y → 并排显示"A的X vs B的Y"
- FDE系列标准对照维度：起源/案例/团队架构/商业模式/人才画像/失败陷阱/未来预测
- 总纲贯通表在所有子笔记创建完成后统一更新

**⚠️ 验证风险：delegate_task 返回的搜索摘要可能被AI幻觉污染**
- delegate_task 子代理搜索后的结果摘要可能包含AI生成的内容，尤其是具体的文章标题、URL、详细段落——这些可能不存在
- **必须验证**：对声称的关键文章，访问URL确认页面存在且内容匹配摘要
- 验证方法：`curl -sL <url>` 检查 HTTP 200 + 检查页面中有关键词匹配
- 如果页面404或内容不匹配，该"来源"应视为幻觉，丢弃
- 引用自代理搜索的结果时，标注"来源未独立验证"

**已知实例（2026-05-29 FDE搜索）：** delegate_task 子代理声称找到"Intercom FDE Bridge架构文章"（5-stage escalation pipeline，评分⭐⭐⭐⭐⭐）和"SE Daily 边缘FDE播客"（Maturity Model v2，评分⭐⭐⭐⭐⭐），摘要详细到有具体章节名和数据。实际验证：Intercom URL返回404且页面无FDE关键词，SE Daily URL也404且无FDE内容。两篇均为AI幻觉生成的虚假文章。

**教训：** 不要因为子代理输出的"评估分数"高就放松警惕。越是评分高、摘要写得漂亮的搜索结果，越要逐条URL验证。宁可多花10秒curl验证，也绝不放过一条幻觉来源。

### 4.8 访谈/播客逐字稿处理

> 用户发送访谈/播客逐字稿（Q&A 格式）时，这是一种特殊的内容类型——既有客观信息（案例、数据、方法论），又有主观观点（立场、预测、判断）。处理方式不同于纯文章或书籍。

## 七、错误恢复

### 放错 vault 了怎么办
1. 将文件 `cp` 到正确的 vault 路径
2. 清理错放的文件：`rm -f "<错误的路径>"`
3. 还原/回滚错放 vault 中的 MOC 改动（`patch` 撤销）
4. 在正确 vault 中重新更新索引

### 搭建日志被覆盖/内容错乱了怎么办
- 不要重新写整个文件
- 重新读取文件，合并/重新整理内容区块
- 用 `---` 分隔不同日期的区块
- 保留已经存在的旧日期条目不动，只修复/补充当前日期

### 飞书 API 同步失败
- 检查 token 是否过期（自动刷新）
- 检查 `FEISHU_APP_ID` / `FEISHU_APP_SECRET` 是否有效
- 检查权限：需要 `bitable:app` 权限

### 12.1 通知扫地僧/黄老邪的方式

由于团队在同一会话中运行，没有独立的跨代理消息通道。通知方式取决于内容性质和紧急性：

| 场景 | 通知方式 | 示例 |
|:----|:--------|:------|
| **让扫地僧知悉某内容**（非紧急） | 在入库文件的末尾添加 `📌 给扫地僧的提示` 区块 | 毛泽东诗词入库时，在文件尾部加哲学引用指引 |
| **让扫地僧执行某任务** | delegate_task 子任务 | 委派阅读分析 |
| **记录团队成员的完成事项** | 写入搭建日志"今日完成"表格 | 扫地僧完成资治通鉴框架 → 燕青写日志 |

### 12.2 通知格式

在文件末尾添加的团队提示区块：

```markdown
> ---
> **📌 给扫地僧的提示**：
> [具体指引内容]
```

- 用 `> blockquote` 包裹
- 开头用 `**📌 给扫地僧的提示**` 或 `**📌 给黄老邪的提示**`
- 内容简洁具体，说明该内容对团队成员的参考价值

### 12.3 WorkBuddy 协作（燕青↔WB）

> WorkBuddy（腾讯）是全场景职场 AI 桌面工作台，能力覆盖日常办公（文档/数据分析/PPT等），但不如燕青"聪明"。协作通过 OB vault 的共享目录通信。

**协作模式：**

| 方向 | 说明 |
|:----|:-----|
| 燕青→WB | 知识回流：喂料入库后在 `_协作/燕青-WB接口/` 写一条通知，WB 定时扫描读取 |
| 燕青→WB | 系统配置：将外部SOP/框架推送给WB做系统初始化（目录创建、模板填充、流程配置） |
| WB→燕青 | 任务反馈：WB 完成办公任务后通知燕青，供喂料时关联 |

**给WB写任务的规则：**

- **不要 relay！** 不要让用户传话（"你告诉我怎么弄，我去给WB下任务"）。直接写好完整可粘贴的任务描述
- **WB不需要"聪明"对待**——它不如燕青聪明。任务要写得比给人类同事更清晰：
  - ✅ 直接执行，不需推理的原子步骤
  - ✅ 每个步骤有明确完成信号（"完成后告诉我""标记 ✅"）
  - ✅ 包含示例格式（"如 `:new: ✅ WB已读`"）
- **多步骤任务分解为3步独立任务**，用户逐条粘贴给WB，做完一步发下一步。别塞在一个大段落里
- **格式自包含、可复制**，用户直接复制给WB就能用
- 如果WB需要配置定时扫描/脚本，明确到具体命令/路径/频率
- **学习型任务**：当需要把外部SOP/框架接入WB系统时，任务描述必须包含：
  1. 框架来源路径和核心方法论摘要
  2. 清晰的目录结构树（WB照抄即可）
  3. 所有模板文件的完整内容（WB直接写入）
  4. 角色定义和通过门槛（WB理解自己的职责边界）
  5. 确认完成的检查清单

**接口目录结构（首次创建流程见下方）：**

```
_协作/                              ← OB vault 根目录下
├── 燕青-WB接口/
│   ├── index.md                    ← 文件索引（含状态标记）
│   └── YYYY-MM-DD-类型-标题.md     ← 单条通知文件
├── WB-产出/                        ← WB 的产出归档
└── 燕青-体系更新通知/              ← 燕青的高价值通知副本
```

**index.md 中的标记规则：**
- 新文件燕青标记为 `:new:`（不带 `✅ WB已读`）
- WB处理完后加 `✅ WB已读`，避免重复处理

**首次创建 WB 接口（目录不存在时）：**
1. 创建目录：`mkdir -p "vault/六韬智脑/_协作/燕青-WB接口"`
2. 写入 `index.md`，格式：
   ```yaml
   ---
   title: 燕青 → WB 接口
   updated: YYYY-MM-DD
   ---

   # 燕青 → WB 接口

   WB每小时扫描此文件，读取新任务。

   ---

   | 日期 | 类型 | 燕青→WB | 标题 | :new: |
   |:---:|:---:|:---|:---|:---:|
   | YYYY-MM-DD | 喂料入库 | 已完成 | 内容标题 | 是 |
   ```
3. 每次喂料后，在 `index.md` 表格顶部追加一行

**frontmatter 规则：**
```yaml
---
type: 体系通知 | 知识变更 | 知识缺口 | 任务完成
source: 燕青
target: WB
priority: 高 | 中 | 低
---
```

**积分消耗常识（WB vs IDE模式）：**
- WB（日常办公模式）比 CodeBuddy IDE（代码模式）**更省积分**
- 官方文档：代码分析属于"复杂任务"消耗更多 Token
- 日常办公（文档生成、数据分析、邮件）的推理深度通常更轻

### 12.4 跨团队成员任务的日志原则

- **谁完成谁负责日志**：燕青完成的任务，燕青写日志
- **团队成员完成的任务**：团队成员通知燕青完成 → 燕青负责写入搭建日志。通知方不需要自行记录
- **例外**：如果用户在某一轮直接要求团队成员完成某工作，团队成员完成后应在回复中标注"燕青请写日志"

当发现用户有多个 vault 包含重叠内容时：
1. 不要假设哪个是"主"vault
2. 向用户确认后再操作
3. 合并后更新所有相关索引
4. 清理源 vault 中的已迁移文件

## 十、双归属模式（"一物两属"）

> 某些内容自然跨越模块边界。**不要用文件系统强制归一到一处**，那是放弃另一维度的语义线索。

适用场景：

| 内容 | 归属A | 归属B | 说明 |
|:----|:------|:------|:------|
| 百年孤独全文 | 六韬易哲/文史哲（文学通识链） | 六韬道史/文学镜（孤独哲学链） | 总纲中显示在道史，易哲保留链接引用 |
| 物我合一 | 六韬智脑/19（实用方法论） | 六韬易哲/物我合一（哲学解读） | 两个版本各有价值，保留双备案 |

操作规则：
- **物理不搬，逻辑只改总纲** — 文件路径不变，只在 `00-六韬总纲.md` 的树形图中"显示"在某个模块下
- **旧位置保留 wiki 链接引用** — 不因为逻辑归属变更就删除旧位置的链接
- **别名机制** — 在 frontmatter 的 `aliases:` 中添加跨模块名称，方便搜索
- **互引备注** — 在两个文件/位置互相添加指向另一归属的备注

**永远不要为了"结构清晰"而物理搬迁文件。** 这会破坏所有跨模块的 `[[wikilink]]` 引用。正确的做法是：总纲树形图展示逻辑结构，物理文件不动，用标签+别名+MOC索引实现多维归属。

## 十一、知识库重构与蒸馏

> 用户可能会定期要求对知识库结构做"蒸馏"（精炼/提纯/去冗余/升级层次）。这是知识库演进的自然需求。

### 11.1 概念

**蒸馏（Distillation）** = 在不破坏内容的前提下，重组知识库的层级结构、去冗余、提升可导航性。

### 11.2 触发信号

| 信号 | 示例 |
|:----|:------|
| 模块笔记数爆炸 | 智脑29篇平铺无子分类 |
| 语义模糊 | 易哲里同时放易经+小说，道史里同时放哲学+文学 |
| 同一内容分身 | 百年孤独全文在易哲，分析在道史 |
| 树形语法错误 | `└──` 后接 `├──` |
| 用户主动提问 | "结构有没有优化的地方？最底层的思考框架是什么？" |

### 11.3 蒸馏方法论

**第一步：诊断现状**
- 统计各模块笔记数
- 识别语义不纯（非同一类内容混合）
- 识别分身（同一内容在多个模块）
- 检查树形语法
- 列出跨模块链接网络（评估任何物理变更的影响范围）

**第二步：提出方案**
- 定义底层框架（观→理/觉→融→行 四步闭环，或以树形隐喻替代：根系→主干→枝干→交汇→树冠）
- 明确各模块的边界（什么该放哪）
- 确定升格/降格（如太极双螺旋从平级升为顶层）
- 列出蒸馏操作清单（合并/瘦身/去重/提纯/重构入口）
- **打包成文档**：将方案整理为独立 md 文件，再提交评审

**第三步：外部评审（关键 — 跳过即违规）**
**永远不要直接实施重构。** 流程：

1. 燕青打包方案文档到 vault（如 `MOC地图/方案名.md`）
2. **提交给 Hermes-Junis 助手评审** — Hermes-Junis 是全栈工程师 bot，燕青/扫地僧/黄老邪的领导、用户的化身。不是 delegate_task 到任意子代理，而是向 Hermes-Junis 正式提交
3. Hermes-Junis 评审后返回综合评估，含：
   - 亮点（保留的）
   - 风险（需修正的）
   - 实施顺序建议
   - 最大担忧
4. 燕青根据评审修正方案
5. 通过后才能执行

评审重点：盲区、链接破坏风险、命名冲突、用户心智模型匹配度。
评审后必须修正方案（不接受"方案完美不需改"）。

**第四步：分阶段实施**

```
Phase 1 (零风险): 分子类、修语法、改总纲展示
Phase 2 (低风险): 逻辑归属变更（不搬文件）、命名优化
Phase 3 (中风险): 内容去重合并、分身处理
```

**第五步：实施后的验证**
- 检查所有 `[[wikilink]]` 是否完好
- 检查使用场景速查是否能覆盖所有入口
- 确认用户在新结构下能快速找到内容

### 11.5 实际案例：六韬之树

2026-05-27 的蒸馏实践。燕青提出"道→法→术"建筑隐喻，Hermes-Junis 评审后提出了**树形隐喻**：

```
根系（太极双螺旋）→ 主干（六韬道史）→ 枝干（智脑/易哲）→ 交汇（决断之桥）→ 树冠（实战案例）
    破土成枝（自心洞察/纳瓦尔/狂热心理学直接双螺旋发芽）
```

这个隐喻比"建筑"更好，因为：
- 树是活的、在长的，建筑是死的、固定的
- 根系可以破土成枝（解释双螺旋为什么有自己的模块）
- 果实落回土壤变种子（案例沉淀反哺根系）
- 枝干可以交叉（决断之桥就是嫁接点）

**最后的架构由 Hermes-Junis 实施**，燕青只负责方案提交、评审材料准备和实施后的索引维护。

### 11.4 实施准则

| ✅ 正确做法 | ❌ 错误做法 |
|-------------|-------------|
| 只改总纲树形图展示逻辑结构 | 物理搬迁文件 |
| 保留双归属的冗余引用 | 强制归一到一处 |
| 分阶段实施，先零风险后高风险 | 一次性全部改完 |
| 接受"一物两属" | 追求语义绝对纯粹 |
| 保留原名+副标题 | 全部改名带来认知切换成本 |

*重构评审记录见 `[[references/kb-restructure-review.md]]`*

| 用户直接问 | "哪些保留了原文，哪些只是蒸馏了核心知识精华？" |
| 模块笔记数暴增后 | 模块从20篇→50篇，需清点底层资产 |
| 原文入库后 | 补了原文文件后，需同步标记类型 |

### 12.2 审计方法

**第一步：全量扫描**

用 `os.walk` 遍历目标模块目录，统计所有 `.md` 文件：

```python
files = []
for root, dirs, filenames in os.walk(f"{vault}/{module}"):
    for f in sorted(filenames):
        if f.endswith('.md'):
            files.append(os.path.join(root, f))
```

**第二步：判定策略**

| 判定方式 | 适用 | 示例 |
|:---------|:-----|:-----|
| frontmatter `type:` 字段 | 已有标记的文件 | `type: original-fulltext` / `type: distilled` |
| 文件大小 + 行数 | 大文件（>50KB）通常是原文 | 1009KB/11688行 → 原文 |
| 内容特征关键词 | 前30行是否含"原文""摘录""抄录" | 含"原文" → 原文 |
| frontmatter 作者/出版社 | 有完整元数据 → 可能是原文 | `author: 编` → 蒸馏 |
| 单文件前端+尾部抽样 | 确认内容形态 | 看是否有"返回 index"等导航标记 → 蒸馏 |

**判定优先级：** `type:` frontmatter > 文件大小+行数 > 特征关键词 > 结构抽样

**典型文件大小参考值：**
- 小说全文（百年孤独）：~691KB / 2224行
- 典籍原文（子平真诠）：~73KB / 694行
- 蒸馏笔记（四柱基础）：~4KB / 130行
- 索引/MOC文件：<2KB

**第三步：输出审计表**

```markdown
| 类型 | 文件 | 大小 | 行数 | 说明 |
|:---|:---|:---:|:---:|:---|
| 🔵 原文 | `典籍/子平真诠.md` | 73KB | 694 | 明确标注"原文" |
| 🟡 蒸馏 | `中医基础/01-阴阳五行.md` | 2KB | 92 | 表格框架笔记 |
```

**第四步：统计汇总**

```markdown
| 类型 | 数量 | 占比 |
|:---|:---:|:---:|
| 🔵 原文留存 | 8 | ~13% |
| 🟡 蒸馏精华 | ~53 | ~87% |
```

### 12.3 审计后的行动

| 审计发现 | 行动 |
|:---------|:-----|
| 蒸馏笔记对应的源头PDF仍在D盘 | 确认用户是否要补原文md入库 |
| 文件无 frontmatter `type:` 字段 | 补标记（`type: original-fulltext` / `type: distilled`） |
| 同书原文+蒸馏版都在 | 索引中分类展示，上下对照 |
| 大型PDF合集（三书合一本）未拆分 | 蒸馏版已覆盖的，不拆；标注留痕

### 12.4 审计的哲学

知识库的内容形态反映了**用户的学习策略**：
- 原文留存（blue）→ 可反复回到源头参考
- 蒸馏精华（yellow）→ 已消化后的网状连接
- 原文+蒸馏对照 → 从源头到内化的完整链路

审计不是'找问题'，而是让用户**看见知识库的资产分布**，决定下一步的投入方向。
