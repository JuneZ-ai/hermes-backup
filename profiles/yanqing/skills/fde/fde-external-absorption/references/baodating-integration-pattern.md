# 包打听集成模式 — 吸收产物如何接入下游产品

> ⚠️ 核心原则：**现有 bot 团队即是产品入口。** 不为任何产品搭独立飞书 bot、Web 前端或 API 网关。六韬 Bot 团队（燕青/扫地僧/黄老邪/Hermes-Junis）就是用户与企业知识互动的前端。

外部技术吸收的最终价值在于**产物被下游产品使用**。
包打听（企业经验引擎）是 FDE 的核心产品，吸收产物的集成模式如下。

## 集成架构

```
用户提问（飞书对话）
    │
    ▼
六韬 Bot 团队 ← 用户入口，就是这里
    │─ 燕青（执行检索）
    │─ 扫地僧（哲学思辨）
    │─ 黄老邪（分析研究）
    │─ Hermes-Junis（全栈/评审）
    │
    ▼
┌──────────────────────┐
│  后端组件             │  ← 吸收产物在这里
│  ┌──────────────────┐│
│  │ hybrid-memory    ││  ← REST 8421（BM25+向量+RRF）
│  │ Mermaid 压缩     ││  ← 深度分析报告
│  │ 渐进上下文注入    ││  ← 回答策略
│  │ 用户画像         ││  ← 企业画像
│  └──────────────────┘│
└──────────────────────┘
```

**一句话**：bot 是入口，吸收产物是后端。不建独立前端。

## 本次吸收的集成映射

| 吸收产物 | 下游用途 | Bot 调用方式 |
|:---------|:---------|:-------------|
| hybrid-memory MCP | 包打听检索层 | 燕青调 REST 8421 搜知识库 → 回答案 |
| Mermaid 压缩 | 包打听深度分析报告 | Agent 长任务中自动触发 |
| 渐进式上下文注入 | 包打听回答生成策略 | 燕青按 L3→L2→L1 逐层加载 |
| 用户画像 | 包打听企业客户画像 | 燕青调 auto-profile 生成→人确认 |

## 产品接入步骤模板

1. **确定需求**：用户/企业需要什么能力？
2. **匹配已有产出**：哪个 skill/MCP/脚本解决了这个需求？
3. **验证 Bot 能否直接调用**：燕青等 bot 是否能通过现有工具链调用？
4. **封装适配**：MCP 需要 REST 包装？脚本需要参数化？
5. **测试**：在现有对话窗口中跑通（不依赖新入口）
6. **归档**：更新产品架构文档 + 搭建日志

## 反例

- ❌ 搭独立飞书 bot：「包打听-bot」
- ❌ 搭 Web 前端：「包打听查询页面」
- ❌ 搭 API 网关

## 正例

- ✅ 用户问「客户投诉怎么处理」→ 燕青搜 hybrid-memory → 返回答案
- ✅ 用户问「分析这个企业的战略」→ 黄老邪分析 → 扫地僧思辨 → Hermes-Junis 评审
- ✅ 用户问「帮我查一下供应商评估标准」→ 燕青搜索引 → 返回历史记录
